# PgKronika

[English version](README.md)

PgKronika — экспериментальная система наблюдения за PostgreSQL. Она хранит
историю диагностики в неизменяемых локальных сегментах, а не в центральной базе
метрик.

Коллектор работает на сервере базы данных. Он читает данные PostgreSQL,
операционной системы, cgroup и журналов, а затем пишет самодостаточные
PGM-сегменты. Отдельный веб-процесс открывает эти сегменты через интерфейс,
MCP и JSON API. Если включена удалённая архивация, её выполняет отдельный
процесс: коллектору не нужны облачные ключи.

Проект пока на ранней стадии. Сейчас основная работа идёт вокруг формата файла,
состояния записи, реестра типов и тестовой инфраструктуры. Готового продукта
для мониторинга ещё нет.

## Цели

- Держать свежую диагностическую историю рядом с сервером базы данных.
- Делать каждый сегмент читаемым сам по себе: схемы, смещения, словари, события
  и заранее посчитанные точки графиков хранятся вместе с данными.
- Сохранять подробные PostgreSQL-данные с высокой кардинальностью, не
  превращая их в Prometheus-метрики.
- Поддерживать разбор инцидентов без доступа к исходному серверу: сегменты
  можно скопировать и открыть локально.
- Разделять сбор, чтение, архивацию и диагностику формата по разным
  исполняемым файлам.

## Архитектура

PgKronika планируется как четыре статически собранных исполняемых файла:

| Команда | Роль |
| --- | --- |
| `pg_kronika-collector` | Собирает данные PostgreSQL, ОС, cgroup и журналов; пишет локальные PGM-сегменты. |
| `pg_kronika-web` | Отдаёт веб-интерфейс, MCP и JSON API поверх хранилища сегментов. |
| `pg_kronika-archiver` | Выгружает готовые локальные сегменты, если включена удалённая архивация. |
| `pg_kronika-dump` | Проверяет, извлекает, сравнивает и разбирает сегменты. |

Формат хранения называется PGM. Это неизменяемый сегмент со снимками состояния,
словарями, событиями, данными для графиков и концевым каталогом. Сегменты можно
копировать, проверять и открывать без подключения к исходному серверу базы
данных.

## Принятые решения

- `type_id` секции называет точную схему данных в этой секции. Если в новой
  основной версии PostgreSQL у представления меняются колонки или счётчики
  переезжают в другое представление, PgKronika выделяет новый вариант схемы
  (`C_SSS_VVV`) вместо nullable-колонок "для другой версии". `NULL` означает
  отсутствие значения во время выполнения, а не "этой колонки нет в этой версии
  PostgreSQL".
- Коллектор читает основную версию PostgreSQL один раз из handshake соединения
  и вызывает тот коллектор, чей кодек совпадает со схемой этой версии. Текущие
  разбиения: `1_006_001` для `pg_stat_bgwriter` в PG 15-16, `1_006_002` для
  `pg_stat_checkpointer` и сокращённого `pg_stat_bgwriter` в PG 17+,
  `1_020_001` для контекста сбросов в PG 15 и `1_020_002` для контекста
  сбросов в PG 16+ с `pg_stat_io`.
- Контекст уровня сегмента хранится в служебных секциях. Версии PostgreSQL и
  расширений относятся к `instance_metadata`; времена сбросов — к
  `reset_metadata`; настройки и GUC — к семейству settings, когда оно появится.
  Статистические секции не повторяют этот контекст в каждой строке.
- README отдельных крейтов и rustdoc рядом с реализацией являются контрактной
  документацией. Каталог `docs/` остаётся историей проектирования, пока код
  быстро меняется.

## Структура репозитория

```text
bins/      исполняемые файлы команд
crates/    внутренние Rust-крейты
docs/      история проектирования
xtask/     команды обслуживания репозитория
```

Актуальное поведение описано рядом с кодом: в README отдельных крейтов,
rustdoc и тестах. Каталог `docs/` остаётся историей проектирования, пока
реализация быстро меняется.

## Разработка

Установите версию Rust, указанную в
[`rust-toolchain.toml`](rust-toolchain.toml), затем запустите:

```sh
cargo check --workspace
cargo run -p xtask -- check-deps
```

`check-deps` проверяет архитектурные зависимости между исполняемыми файлами и
внутренними крейтами. Например, код хранилищ не должен попадать в зависимости
коллектора, а код PostgreSQL-источника не должен попадать в зависимости
веб-процесса.

BDD-проверку для PostgreSQL 15, 16 и 17 можно запускать локально, в GitHub
Actions и в GitLab CI. Команды для быстрой проверки, запуска через
Docker/Buildx, запуска через Nix и примеры CI описаны в
[crates/kronika-bdd/README.ru.md](crates/kronika-bdd/README.ru.md).

Реализация движется по крейтам: низкоуровневый формат, состояние записи, реестр
типов, источники данных, затем серверные исполняемые файлы.
