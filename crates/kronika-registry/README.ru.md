# kronika-registry

[English version](README.md)

`kronika-registry` задаёт, что означает каждый `type_id` секции, и содержит
кодеки для тел секций. В крейте есть:

- формат `type_id` и `SectionClass`;
- контракт типа: колонки, классы колонок, ключ сортировки, семантика сбора;
- проверка реестра, которая ловит нарушения контрактов;
- трейт `Section` и `#[derive(Section)]`, который его имплементит из структуры
  (контракт и кодек типа);
- общий рантайм кодека и `decode_any` для диспетча по `type_id`.

## Идентификаторы типов

`type_id` — это `C_SSS_VVV` в десятичном виде: класс `C`, источник `SSS`
внутри класса, версия формата `VVV` (с 001). `1_006_001` — класс 1
(снимок), источник 006, версия 001; графики используют двузначный класс 10,
поэтому `10_001_001` — класс 10, источник 001, версия 001.

`TypeId` разбирает число (`class_digit`, `source`, `version`,
`section_class`). `TypeId` создаётся только внутри этого крейта, через
`#[derive(Section)]`: проверяющий конструктор отвергает неизвестный класс или
нулевой источник либо версию, а поскольку id контракта лежит в `const`,
невалидный `#[section(id = ...)]` — это ошибка компиляции. Каждый id в реестре
валиден по построению, и ни один внешний крейт не может его создать.

`SectionClass` — это `Snapshot` (1), `Event` (2), `Dictionary` (3),
`Chart` (10).

## Контракт типа

`TypeContract` — запись реестра для одного `type_id`:

- `columns` — записи `Column` в порядке схемы: имя, `ColumnType`,
  `ColumnClass` и признак того, что колонка может быть `NULL`;
- `sort_key` — имена колонок по порядку; каждое имя должно быть колонкой;
- `semantics` — как собирается источник;
- `name`, `deprecated`.

`ColumnClass` (в README крейта формата это классы C/G/L/T):

- `Cumulative` — монотонный счётчик; скорости считаются как дельты во времени;
- `Gauge` — мгновенное значение;
- `Label` — идентичность или атрибут сущности;
- `Timestamp` — `i64`, микросекунды Unix.

`ColumnType` — тип значения на диске (`I64`, `F64`, `U64`, `Bool`, `Ts`).
`Semantics` — это `SnapshotFull`, `ConditionalFull`, `EventStream`,
`Changed` или `OnChange`.

## Проверка реестра

`lint_registry` проверяет каждый известный тип и запускается в CI через тесты
крейта. Валидность самого id (класс, источник, версия) обеспечивается при
конструировании `TypeId`, поэтому линтер покрывает лишь то, что конструктор
не может, — правила уровня контракта или всего реестра. Ошибкой считается:

- два контракта с одинаковым `type_id` (id не переиспользуются);
- имя в ключе сортировки, которого нет среди колонок;
- тип `Changed` без колонки `is_baseline`;
- колонка класса `Timestamp`, если это не обязательная колонка `ts` без
  `NULL`.

## Секции снимков

Тело секции снимка — Parquet-файл со сжатием zstd. `arrow_schema` строит
Arrow-схему из колонок контракта, поэтому кодеки используют тот же порядок
колонок, типы Arrow и правила `NULL`, которые указаны в реестре.

Источник истины о схеме — реестр, поэтому секция **не** хранит вторую её
копию: код записи пропускает блоб `ARROW:schema`, который arrow-rs пишет по
умолчанию, и обнуляет строку версии Arrow. Это убирает фиксированные ~1.1 КБ
на секцию (около четверти однострочной секции) и делает кодирование
детерминированным. В файле остаётся только нативная Parquet-схема — физические
типы и раскладка колонок, которые нужны декодеру для чтения column chunk'ов.

Декодер считает контракт авторитетным: он накладывает колонки и типы контракта
и возвращает типизированную ошибку при любом несовпадении. Если байты под
`type_id` не декодируются — значит, код записи нарушил контракт, и читатель
пропускает такую секцию с диагностикой, а не пытается угадать.

## Сгенерированные кодеки

Тип — это структура с `#[derive(Section)]` (`kronika-derive`); структура и
есть единственный код на тип. Derive читает тип каждого поля (тип на диске и
nullable) и `#[column(class)]`, плюс заголовок `#[section(..)]`, и имплементит
трейт `Section`: const-контракт и `encode`/`decode`. Рукописного
по-колоночного кода, который мог бы разойтись с контрактом, нет.

Обрамление, сжатие и все границы памяти живут один раз в общем рантайме кодека
(`encode_section` / `decode_section`); сгенерированный код лишь даёт по одному
билдеру/ридеру на колонку, поэтому новый тип не может забыть границу. Типы
колонок мапятся на самый узкий тип Arrow, который вмещает значение (`i32` →
32 бита, а не 64), чтобы секции были маленькими. `kronika-derive` — это ровно
тот крейт, который планирует `architecture.md` §7: структура → схема, кодер,
декодер.

## Трейт Section

Каждый сгенерированный кодек — это `impl Section for T`, поэтому обобщённый код
работает через `T: Section` (`CONTRACT`, `encode`, `decode`), не называя каждый
тип, — общий roundtrip-тест пишется один раз, а не на каждый тип.

При чтении сегмента `type_id` — это `u32` времени выполнения, поэтому читатель
не может назвать `T`. `decode_any(type_id, bytes)` диспетчеризует через
`registry()` и возвращает `DecodedSection` — Arrow-пакеты (`RecordBatch`) плюс
`DecodeStats` (входные байты, группы строк, пакеты, строки), которые читатель
экспортирует как RED-метрики, — проверенные против контракта. Ему не нужен
конкретный тип и не нужен `match` на тип, поэтому новый тип стоит одной записи
в `registry()` и сразу декодируется — именно это позволяет реестру вырасти до
сотен типов без проводки на каждый тип.

`decode` и `decode_any` принимают владеемые `Bytes` и никогда не копируют
секцию — Parquet-reader слайсит её на месте. Читатель держит сегмент один раз
(mmap или одно чтение) и отдаёт каждой секции zero-copy-слайс; потоковый
источник может переиспользовать буферы через `BytesPool`, который возвращает
буфер себе, когда декодированный `Bytes` дропается, — поэтому ровный цикл
декода не аллоцирует на каждую секцию. Пул покрывает только входной буфер;
разжатые и Arrow-данные по своей природе свежие.

Трейт `Section` публичный, а макрос `#[derive(Section)]` — нет. Все типы секций
живут в этом крейте (derive проводит id через крейт-приватный конструктор
`TypeId`), поэтому макрос — внутренний инструмент реестра, а не публичная точка
расширения.

## Пример: тип `1_006_001`

Типы не документируются по секции на id в этом README — каталог типов это
контракты реестра в коде. `1_006_001` оставлен здесь как единственный
разобранный пример.

`pg_stat_bgwriter` + `pg_stat_checkpointer`, однострочный снимок
(`SnapshotFull`, ключ сортировки `(ts)`). В PostgreSQL 17 часть счётчиков
`pg_stat_bgwriter` перенесена в `pg_stat_checkpointer`; коллектор читает оба
представления и пишет одну строку. Колонки, которых нет в PostgreSQL 17
(`buffers_backend`, `buffers_backend_fsync`), сохраняются как `NULL`, а не
как `0`.

Весь кодек `1_006_001` — это его аннотированная структура плюс
`#[derive(Section)]`; `BgwriterCheckpointer::encode` / `decode` переводят срез
строк в тело Parquet-секции и обратно.

## Границы памяти

Секция снимка содержит не больше `MAX_SECTION_ROWS` строк. `encode` отвергает
входной срез большего размера. `decode` проверяет секцию до чтения строк:
размер тела должен укладываться в `MAX_SECTION_BYTES`, число групп строк — в
`MAX_ROW_GROUPS`, а число строк из метаданных и после декодирования — в
`MAX_SECTION_ROWS`.

Пик памяти в `encode` — срез строк вызывающего кода плюс одна группа строк
Parquet (`row group`). `decode` принимает владеемые `Bytes` и не копирует
секцию — Parquet-reader слайсит её на месте. `decode_section` стримит, по
одному пакету чтения за раз, поэтому его добавочный пик — один пакет;
`decode_batches` (и `decode_any`) возвращает всю секцию как Arrow-пакеты,
поэтому его добавочный пик — разжатые строки секции. Оба ограничены
`MAX_SECTION_ROWS` теми же caps, намного выше того, что текущие однострочные
источники набирают за один сегмент при обычном опросе; они ловят ошибки записи
и некорректные секции, а не нормальные данные.

Один риск остаётся в самом декодировании Parquet: страница допустимого размера
может запросить большой буфер по своему заголовку, и байтовый cap этого не
ограничивает. Поэтому `decode` исходит из того, что байты уже CRC-проверены:
сканер партов (`kronika-format` `validate_part`) сверяет CRC каждой секции с
каталогом до того, как её байты находят, — обычный путь чтения покрыт. Тулза,
декодирующая сырые байты, обязана проверить сама. Это ловит порчу носителя;
полностью подделанные сегменты — за пределами модели защиты формата.

## Тесты

- разбор и проверка `type_id`;
- проверка реестра, отдельный тест на каждую ошибку;
- кодек `1_006_001`: полный цикл `encode`/`decode` без потери значений,
  PG16- и PG17-строки в одной секции, `NULL` остаётся отличимым от нуля,
  тело имеет формат Parquet-файла, ограничения строк, байтов и групп строк
  срабатывают. Эталонный тест сравнивает декодированные значения, а не сырые
  байты Parquet: байты зависят от метаданных и кодировок в текущей версии
  Arrow;
- обобщённый путь: каждый зарегистрированный тип проходит полный цикл на пустой
  секции через `decode_any` без кода на тип; `decode_any` сообщает статистику
  декода и отвергает незарегистрированный `type_id`.

## Что дальше

`#[derive(Section)]` сможет генерировать ещё и эталонный roundtrip-тест на тип,
когда типов станет несколько. Новые типы появятся вместе со следующими
источниками данных. Фильтры Блума и настройка групп строк Parquet нужны не
здесь, а в первых типах с большим числом уникальных значений.
