# kronika-registry

[English version](README.md)

`kronika-registry` задаёт, что означает каждый `type_id` секции, и содержит
кодеки для тел секций. В крейте есть:

- формат `type_id` и `SectionClass`;
- контракт типа: колонки, классы колонок, ключ сортировки, семантика сбора;
- проверка реестра, которая ловит нарушения контрактов;
- трейт `Section` и `#[derive(Section)]`, который строит из структуры контракт
  и кодек типа;
- общий код кодеков и `decode_any`, который выбирает декодер по `type_id`.

## Идентификаторы типов

`type_id` — это `C_SSS_VVV` в десятичном виде: класс `C`, источник `SSS`
внутри класса, версия формата `VVV` (с 001). `1_006_001` — класс 1
(снимок), источник 006, версия 001; графики используют двузначный класс 10,
поэтому `10_001_001` — класс 10, источник 001, версия 001.

`TypeId` разбирает число (`class_digit`, `source`, `version`,
`section_class`). `TypeId` создаётся только внутри этого крейта, через
`#[derive(Section)]`. Проверяющий конструктор отвергает неизвестный класс,
нулевой источник и нулевую версию. Id контракта записан в `const`, поэтому
ошибка в `#[section(id = ...)]` останавливает компиляцию. Каждый id в реестре
корректен по построению, а внешний крейт не может создать новый `TypeId`.

`SectionClass` — это `Snapshot` (1), `Event` (2), `Dictionary` (3),
`Chart` (10).

## Контракт типа

`TypeContract` — запись реестра для одного `type_id`:

- `columns` — записи `Column` в порядке схемы: имя, `ColumnType`,
  `ColumnClass` и признак того, что колонка может быть `NULL`;
- `sort_key` — имена колонок по порядку; каждое имя должно быть колонкой;
- `semantics` — как собирается источник;
- `name`, `deprecated`.

`ColumnClass` (в README крейта формата это классы C/G/L/T):

- `Cumulative` — монотонный счётчик; скорости считаются как дельты во времени;
- `Gauge` — мгновенное значение;
- `Label` — идентичность или атрибут сущности;
- `Timestamp` — `i64`, микросекунды Unix.

`ColumnType` — тип значения на диске: целочисленные и плавающие базовые типы
(`I8`…`I64`, `U8`…`U64`, `F32`/`F64`), `Bool`, `Ts` (`i64`, временная метка) и `StrId`
(`u64`-ссылка в словарь строк сегмента — байты лежат там, не в секции). Тип
колонки задаёт Rust-тип поля: поле `Ts` или `StrId` отображается в этот тип, а
поле `Option<T>` означает, что колонка может хранить `NULL`.
`Semantics` — это `SnapshotFull`, `ConditionalFull`, `EventStream`,
`Changed` или `OnChange`.

## Проверка реестра

`lint_registry` проверяет каждый известный тип и запускается в CI через тесты
крейта. Валидность самого id (класс, источник, версия) обеспечивается при
конструировании `TypeId`, поэтому линтер покрывает лишь то, что конструктор
не может, — правила уровня контракта или всего реестра. Ошибкой считается:

- два контракта с одинаковым `type_id` (id не переиспользуются);
- имя в ключе сортировки, которого нет среди колонок;
- тип `Changed` без колонки `is_baseline`;
- колонка класса `Timestamp`, если это не обязательная колонка `ts` без
  `NULL`.

## Секции снимков

Тело секции снимка — Parquet-файл со сжатием zstd. `arrow_schema` строит
Arrow-схему из колонок контракта, поэтому кодеки используют тот же порядок
колонок, типы Arrow и правила `NULL`, которые указаны в реестре.

Перед записью `encode` сортирует строки по `sort_key` контракта, чтобы соседние
значения в колонке были похожи и хорошо сжимались. Порядок строк с равным ключом
не определён, и декод возвращает строки в этом отсортированном порядке, а не в том,
в котором они были переданы.

Схему задаёт реестр, поэтому секция **не** хранит вторую копию схемы. Код
записи пропускает запись `ARROW:schema`, которую arrow-rs добавляет по
умолчанию, и очищает строку версии Arrow. Это убирает фиксированные ~1,1 КБ
на секцию (около четверти однострочной секции) и делает байты стабильными. В
файле остаётся штатная схема Parquet: физические типы и раскладка колонок,
нужные для чтения колоночных блоков.

При чтении файл сверяется с контрактом. Если колонки, типы или правила `NULL`
не совпадают, декодер возвращает типизированную ошибку. Если байты под
известным `type_id` не декодируются, код записи создал плохую секцию. Читатель
может пропустить её и показать диагностику, не пытаясь угадывать схему.

## Кодеки из derive

Тип — это структура с `#[derive(Section)]` (`kronika-derive`). Derive читает
Rust-тип каждого поля (тип на диске и возможность `NULL`), атрибут
`#[column(class)]` и заголовок `#[section(..)]`. Из этого он реализует
`Section`: const-контракт и функции `encode`/`decode`.

Код чтения и записи колонок строится из той же структуры, которая задаёт
контракт. Общий код `encode_section` / `decode_section` отвечает за формат
Parquet, сжатие и лимиты памяти. Код, созданный derive, передаёт только один
построитель или читатель на поле, поэтому новый тип проходит через те же
проверки, что и существующие. Типы колонок сопоставляются с самым узким типом
Arrow, который вмещает значение (`i32` → 32 бита, а не 64), чтобы секции не
раздувались.

## Трейт Section

Каждый кодек из derive — это `impl Section for T`. Обобщённый код получает
`CONTRACT`, `encode`, `decode` и `ts_range`, не перечисляя все типы реестра.
На этом держатся общий тест полного цикла и буферы строк в `kronika-writer`.

При чтении сегмента известен только `type_id`. `decode_any(type_id, section)`
находит контракт через `registry()` и возвращает Arrow-пакеты (`RecordBatch`) и
`DecodeStats`. Ошибки этого пути сохраняют тот же `type_id`, поэтому метрики
чтения могут использовать одну метку для успеха и отказа. Новый тип начинает
читаться после одной записи в `registry()`. `T::decode` и `decode_any` одинаково
проверяют набор колонок, порядок, типы и правила `NULL`.

`decode` и `decode_any` принимают `VerifiedSection`: владеемые `Bytes`, чей CRC
сверен с каталогом. Секция не копируется повторно; Parquet читает из этих
байтов. Источник, который читает секции по одной, может использовать
`decode_pooled`: он заполняет буфер из `BytesPool`, а буфер возвращается в пул
после удаления последней ссылки на `Bytes`. Пул закрывает только входной буфер;
разжатые страницы и массивы Arrow всё равно выделяются заново.
`BytesPool::stats()` отдаёт счётчики переиспользования и сброса. Если
`dropped_oversize_total` растёт, `buffer_limit` ниже реальных размеров секций и
пул больше не переиспользует такие буферы.

Трейт `Section` публичный, но закрыт для внешних реализаций: у него есть
закрытый супертрейт, который реализует только derive внутри этого крейта.
Поэтому `T: Section` всегда означает тип из реестра. Сам derive-макрос наружу
тоже не экспортируется.

## Пример: тип `1_006_001`

Типы не документируются по секции на id в этом README — каталог типов это
контракты реестра в коде. `1_006_001` оставлен здесь как единственный
разобранный пример.

`pg_stat_bgwriter` + `pg_stat_checkpointer`, снимок с одной строкой
(`SnapshotFull`, ключ сортировки `(ts)`). В PostgreSQL 17 эти представления
переработаны: счётчики чекпойнтов перенесены и переименованы в
`pg_stat_checkpointer`, добавлены restartpoint-счётчики для hot standby, убраны
`buffers_backend` /
`buffers_backend_fsync`. Коллектор читает оба представления и пишет одну строку;
каждое поле сохраняет стабильное имя и документирует свой источник по версиям, а
счётчик, которого нет в текущем сервере, пишется как `NULL`, а не `0` — в любую
сторону.

Весь кодек `1_006_001` — это его аннотированная структура плюс
`#[derive(Section)]`; `BgwriterCheckpointer::encode` / `decode` переводят срез
строк в тело Parquet-секции и обратно.

## Служебные секции

Два типа обязательны в каждом сегменте с данными PostgreSQL или ОС. По ним
читатель понимает остальные секции:

- `instance_metadata` (`1_021_001`) — отпечаток экземпляра: `pg_version_num`,
  `pg_system_identifier`, hostname и константы ОС (`clock_ticks_per_sec`,
  `page_size_bytes`, `boot_id`, `btime`), которые нужны для чтения ОС-секций без
  внешней настройки.
- `reset_metadata` (`1_020_001`) — общий контекст сбросов: время старта
  postmaster, времена сброса представлений, у которых ещё нет своей секции, версии
  расширений и GUC (`track_io_timing`, `track_wal_io_timing`, `compute_query_id`),
  от которых зависит наличие и смысл колонок.

Сброс статистики работает по представлению:
`pg_stat_reset_shared('bgwriter')` сбрасывает только `pg_stat_bgwriter`. Сброс
может случиться посреди сегмента, поэтому представление со своим `stats_reset`
пишет его в каждой строке собственной секции (`bgwriter_stats_reset` в
`1_006_001`). Так код сравнения видит сброс между двумя замерами.
`reset_metadata`, одна строка на сегмент, такой сброс пропустила бы.

Поэтому один кодек покрывает все версии PostgreSQL. Различия версий записаны
как колонки с `NULL` (`buffers_backend` = `None` на PG17+), а их смысл берётся
из служебных секций, не из нового `type_id` на каждый релиз. `pg_version_num`
говорит, что колонка удалена в PG17; `checkpointer_stats_reset` в строке
(`NULL` до PG17) подтверждает это. Версия источника записана один раз на
сегмент, а не повторяется колонкой в каждом типе.

## Ограничения памяти

Секция снимка содержит не больше `MAX_SECTION_ROWS` строк. `encode` отвергает
срез большего размера до того, как строит хоть одну колонку, и отвергает готовое
тело больше `MAX_SECTION_BYTES` — тот же байтовый лимит, что и при `decode`, —
чтобы писатель не сохранил секцию, которую читатель отклонит. `decode` проверяет
секцию до чтения строк: размер тела должен укладываться в `MAX_SECTION_BYTES`,
число групп строк — в `MAX_ROW_GROUPS`, а число строк из метаданных и после
декодирования — в `MAX_SECTION_ROWS`. `decode_pooled` сверяет заявленную в
каталоге длину с байтовым лимитом до чтения, чтобы повреждённая длина не успела
выделить память.

Пик памяти в `encode` — срез строк вызывающего кода плюс одна группа строк
Parquet (`row group`). `decode` принимает владеемые `Bytes` и не копирует
секцию повторно: Parquet читает из переданных байтов. `decode_section` читает
по одному пакету за раз, поэтому его добавочный пик — один пакет;
`decode_batches` (и `decode_any`) возвращает всю секцию как Arrow-пакеты,
поэтому его добавочный пик — разжатые строки секции. Оба ограничены
`MAX_SECTION_ROWS` теми же лимитами, намного выше того, что текущие однострочные
источники набирают за один сегмент при обычном опросе; они ловят ошибки записи
и некорректные секции, а не нормальные данные.

Один риск остаётся в самом Parquet: страница может запросить память по своему
заголовку. Поэтому декодер принимает `VerifiedSection`, а не сырые байты.
`VerifiedSection::verify(bytes, expected_crc, crc32c)` проверяет CRC секции до
передачи байтов в Parquet. Это ловит порчу носителя; намеренно перестроенный
сегмент с пересчитанными CRC находится за пределами защиты формата.

## Тесты

- разбор и проверка `type_id`;
- проверка реестра, отдельный тест на каждую ошибку;
- кодек `1_006_001`: полный цикл `encode`/`decode` без потери значений,
  PG16- и PG17-строки в одной секции, `NULL` остаётся отличимым от нуля,
  тело имеет формат Parquet-файла, ограничения строк, байтов и групп строк
  срабатывают. Тест сравнивает декодированные значения, а не сырые
  байты Parquet: байты зависят от метаданных и кодировок в текущей версии
  Arrow;
- обобщённый путь: каждый зарегистрированный тип проходит полный цикл на пустой
  секции через `decode_any` без кода на тип; `decode_any` сообщает статистику
  декодирования и отвергает незарегистрированный `type_id`.

## Что дальше

`#[derive(Section)]` сможет генерировать отдельную фикстуру полного цикла для
каждого типа, когда типов станет несколько. Новые типы появятся вместе со
следующими источниками данных. Фильтры Блума и настройка групп строк Parquet
нужны не здесь, а в первых типах с большим числом уникальных значений.
