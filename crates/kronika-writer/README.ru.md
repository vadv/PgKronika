# kronika-writer

[English version](README.md)

`kronika-writer` хранит состояние, из которого собирается `.pgm`-сегмент.
Источники данных передают сюда строки секций и строки для словарей;
`kronika-format` по-прежнему задаёт байтовую раскладку. Этот крейт решает, когда
данные из памяти становятся частью журнала и когда журнал можно записать как
готовый сегмент.

## Что уже есть

Крейт сейчас экспортирует:

- `SectionBuffers` — буферы строк по типам, кодирующие окно сбора в одну
  PGM-часть;
- `Interner` — интернер строк одного сегмента поверх
  `kronika_format::SegmentDicts`;
- `Journal` — файловый журнал `active.parts`.

## Буферы секций

`SectionBuffers` держит строки секций до записи в часть. `push::<T>(row)`
работает для любого `T: Section`; внутри хранится один буфер без конкретного
Rust-типа на каждый `type_id`. Новый тип из реестра не требует нового поля или
ветки в этом коде.

Буфер одного типа ограничен `MAX_SECTION_ROWS`. Когда он заполнен, `push`
возвращает строку как `Err(row)`: вызывающий код сбрасывает буферы и передаёт
эту строку повторно.

`flush(dict_sections, source_id)` кодирует накопленные строки, добавляет
словарные секции после секций данных и возвращает одну PGM-часть. Секции данных
идут в порядке `type_id`; временной диапазон каталога берётся из строк, у
которых есть timestamp. `None` означает, что записывать нечего. После успешного
сброса буферы данных пусты.

## Словарные секции

`dict::encode(window)` превращает текущее окно интернера в тела секций
`dict.strings` и `dict.blobs`, отсортированные по `str_id`. Эти тела не являются
registry-типами `Section`: в них лежат бинарные значения переменной длины.
Но часть хранит их рядом с секциями данных, поэтому `str_id` из снимков
разрешается внутри того же сегмента.

## Interner

У `Interner` два слоя.

Окно — текущий набор словарей в памяти. В нём лежат полные байты значений,
которые ещё не записаны в `active.parts`.

Для значений, уже записанных в журнал, интернер хранит короткие записи: длину,
префикс SHA-256 и требования к размещению, но не исходный текст. Так повторные
SQL-тексты и планы дедуплицируются без удержания всех байтов в памяти до
завершения сегмента.

`flush_window()` вызывает переданную функцию записи для текущего окна. Если
запись прошла, окно заменяется короткими записями и очищается. Если запись не
прошла, окно остаётся без изменений.

`seal()` возвращает остаток окна и финальные требования к размещению для
значений, уже записанных в журнал. Следующий сегмент начинается с пустого
интернера. `stats()` возвращает размеры словарей для метрик коллектора.

Если интернирование завершилось ошибкой — коллизией или конфликтом размещения,
— окно и короткие записи не меняются.

## Journal

`Journal` добавляет части PGM как кадры `PGMP` и синхронизирует каждый кадр
перед возвратом. При открытии существующего журнала он сканирует файл для
восстановления.

Восстановление читает файл кадр за кадром. Пик памяти — одна часть плюс её
декодированный каталог, небольшой буфер для поиска следующего кадра и 16 байт
на каждую найденную часть. Весь журнал в память не загружается. Тест проверяет,
что потоковый сканер ведёт себя так же, как буферный сканер из
`kronika-format`.

Если последний кадр записан не полностью, файл обрезается до последнего
корректного кадра, затем запись продолжается. Повреждение в середине файла или
повреждение в конце, которое не похоже на незавершённую запись, попадает в
`OpenReport` и остаётся на диске для диагностики.

`append()` записывает один кадр. Размер журнала ограничен
`JournalConfig::max_journal_len` (по умолчанию 1 ГиБ). Если следующий кадр
превысит лимит, `append` возвращает `JournalError::Full`; вызывающий код должен
сначала записать текущий журнал как сегмент. `reset()` очищает журнал после
успешной записи сегмента.

## Завершение сегмента

`seal(journal, dest)` записывает части журнала в один неизменяемый сегмент. Он
читает журнал по одной части, копирует тела секций в соседний `*.tmp`, пишет
концевой каталог, делает fsync и публикует файл жёсткой ссылкой. Существующий
файл по `dest` не перезаписывается. Вызывающий код зовёт `Journal::reset`
только после `Ok` от `seal`.

Тип секции может встретиться в нескольких частях. В сегменте это остаётся
несколькими записями каталога; код чтения обрабатывает их по порядку.
Сворачивание в одну отсортированную и заново сжатую секцию — будущая
оптимизация записи, а не изменение формата.

## Что ещё не реализовано

В этом крейте ещё нужно:

- сливать повторные секции одного типа в одну отсортированную секцию;
- убирать дубли обязательных hot-строк в словарях разных частей;
- добавить Bloom-фильтр по `str_id` в `dict.strings`.

Эти изменения не должны менять формат сегмента. Размещение словарей и проверка
кадров журнала остаются в `kronika-format`.
