# kronika-writer

[English version](README.md)

`kronika-writer` отвечает за путь записи коллектора: принимает данные от
источников и доводит их до запечатанного `.pgm`-сегмента. Байтовые структуры
описывает `kronika-format`; этот крейт хранит состояние, которое нужно только
во время записи.

## Что уже есть

Крейт сейчас экспортирует:

- `Interner` — интернер строк одного сегмента поверх
  `kronika_format::SegmentDicts`;
- `Journal` — файловый журнал `active.parts`.

Интернер использует модель словарей из `kronika-format` и добавляет поведение,
которое нужно только коду записи:

- текущее окно хранит полные байты значений, которые ещё не сброшены в журнал;
- карта сброшенных значений хранит только идентичность значения и требования к
  размещению для того, что уже записано в `active.parts`;
- `flush_window()` передаёт текущее окно в callback записи журнала, а затем
  оставляет в памяти только хэши и требования к размещению;
- `seal()` возвращает остаток текущего окна и требования к размещению для
  сброшенных значений, после чего начинает следующий сегмент с пустого
  интернера;
- `stats()` возвращает размеры словарей для собственных метрик коллектора.

Если интернирование завершилось ошибкой — коллизией или конфликтом размещения,
— текущее окно и карта сброшенных значений не меняются.

Журнал добавляет mini-PGM-части как кадры `PGMP`. При открытии существующего
журнала используется сканер восстановления из `kronika-format`:

- оборванный хвост обрезается, после этого запись продолжается;
- повреждение в середине и изолированный хвост попадают в `OpenReport`;
- `append()` записывает один кадр и синхронизирует файл перед возвратом;
- `reset()` очищает журнал после успешного запечатывания.

## Чего здесь пока нет

Буферы по типам, слияние, запечатывание и Parquet-кодирование появятся в
следующих шагах. Правила словарей и байтовая раскладка кадров описаны в
`kronika-format`.
