# kronika-writer

[English version](README.md)

`kronika-writer` хранит состояние, которое нужно коллектору во время записи
сегмента. Крейт находится между источниками данных и готовым `.pgm`-файлом.
`kronika-format` описывает байты на диске; здесь решается, когда записывать
промежуточные данные в журнал и что держать в памяти, пока сегмент собирается.

## Что уже есть

Крейт сейчас экспортирует:

- `SectionBuffers` — буферы строк по типам, кодирующие окно сбора в одну
  PGM-часть;
- `Interner` — интернер строк одного сегмента поверх
  `kronika_format::SegmentDicts`;
- `Journal` — файловый журнал `active.parts`.

## Буферы секций

`SectionBuffers` — окно сбора. Шаг сбора кладёт типизированные строки через
`push::<T>(row)` для любого `T: Section`; буферы держат по одному
type-erased-буферу на `type_id`, а не поле на тип, поэтому новый тип секции стоит
одного `push` и не меняет этот код — именно это нужно сотням типов реестра.
Буфер типа не растёт больше строк одной секции (`MAX_SECTION_ROWS`): полный
буфер возвращает строку как `Err(row)` — сигнал к досрочному flush и повторному
push, так что память ограничена до flush, а не только в `encode`.

`flush(dict_sections, source_id)` кодирует каждый буферизованный тип в
Parquet-тело (`Section::encode`), читает временной диапазон каждого типа
(`Section::ts_range`) и зовёт `kronika_format::build_part`, собирая одну
PGM-часть: data-секции в порядке `type_id`, затем `dict_sections` (раскладка
сегмента COLD-затем-WARM), диапазон времени каталога покрывает буферизованные
строки. Возвращает байты части — или `None`, если ничего нет, — чтобы вызывающий
дописал их в журнал, и очищает data-буферы.

## Словарные секции

`dict::encode(window)` превращает окно интернера в Parquet-тела секций
`dict.strings` и `dict.blobs` — по одной на размещение с записями, отсортированы
по `str_id`. Это не registry-`Section`-типы (их колонки — бинарные переменной
длины), поэтому они кодируются напрямую, но как обычные тела секций, которые
часть несёт рядом с data-секциями, чтобы `str_id`-колонки снимков резолвились в
байты внутри того же сегмента.

## Interner

У `Interner` два слоя.

Окно — текущий набор словарей в памяти. В нём лежат полные байты значений,
которые ещё не записаны в `active.parts`.

Для значений, уже записанных в журнал, интернер хранит короткие записи: длину,
префикс SHA-256 и требования к размещению, но не исходный текст. Так повторные
SQL-тексты и планы дедуплицируются без удержания всех байтов в памяти до
завершения сегмента.

`flush_window()` вызывает переданную функцию записи для текущего окна. Если
запись прошла, окно заменяется короткими записями и очищается. Если запись не
прошла, окно остаётся без изменений.

`seal()` возвращает остаток окна и финальные требования к размещению для
значений, уже записанных в журнал. Следующий сегмент начинается с пустого
интернера. `stats()` возвращает размеры словарей для метрик коллектора.

Если интернирование завершилось ошибкой — коллизией или конфликтом размещения,
— окно и короткие записи не меняются.

## Journal

`Journal` добавляет части PGM как кадры `PGMP` и синхронизирует каждый
кадр перед тем как вернуть управление. При открытии существующего журнала он
запускает восстановление.

Восстановление читает файл потоково, кадр за кадром: пик памяти — одна часть
плюс её декодированный каталог (не больше двух `max_part_len`, и это
достижимо только патологической частью из одного каталога), небольшое
скользящее окно при повторной синхронизации и 16 байт каталога частей на
кадр — но не весь журнал. Эквивалентность потокового сканера и буферного
сканера из `kronika-format` закреплена тестом.

Если последний кадр записан не полностью, файл обрезается до последнего
корректного кадра, затем запись продолжается. Повреждение в середине файла или
повреждение в конце, которое не похоже на незавершённую запись, попадает в
`OpenReport` и остаётся на диске для диагностики.

`append()` записывает один кадр. Размер журнала ограничен
`JournalConfig::max_journal_len` (по умолчанию 1 ГиБ, стартовое значение):
когда добавление превысило бы лимит, возвращается `JournalError::Full` —
это сигнал досрочно слить журнал в сегмент. `reset()` очищает журнал после
успешного завершения сегмента.

## Завершение сегмента

`seal(journal, dest)` сливает части журнала в один иммутабельный сегмент. Он
читает журнал по одной части — пик памяти это одна часть плюс растущий каталог,
а не весь сегмент, — копирует тела секций каждой части в соседний `*.tmp`,
пишет концевой каталог, делает fsync и публикует жёсткой ссылкой, так что
существующий сегмент никогда не перезаписывается. Вызывающий зовёт
`Journal::reset` только после того, как `seal` вернул `Ok`.

Тип секции, встретившийся в нескольких частях, остаётся повторными записями
каталога — это корректные мульти-part секции, которые читатель обрабатывает по
порядку. Сворачивание их в одну отсортированную пересжатую секцию — оптимизация
того же пути; она меняет то, как пишутся тела, а не формат сегмента.

## Что ещё не реализовано

Остаются три оптимизации пути запечатывания, ни одна не меняет формат сегмента:
sort-merge, пересжимающий повторные секции одного типа в одну отсортированную;
слияние словарей, дедуплицирующее strict-hot значение, повторённое в нескольких
частях (в остальном части непересекающиеся, поэтому seal копирует их словари как
есть); и bloom-фильтр по `str_id` в `dict.strings` для межсегментного поиска.
Размещение словарей и проверка кадров журнала описаны в `kronika-format`.
