# Формат сегмента PGM

Версия документа: draft-4, 2026-06-12.

## 1. Назначение

PGM — формат одного иммутабельного сегмента с данными коллектора за короткий
интервал времени, обычно около 10 минут. Сегмент хранит:

- снимки метрик PostgreSQL и ОС: `pg_stat_*`, `/proc`, cgroup, системные
  счётчики;
- события для таймлайна;
- словари строк и больших текстовых значений;
- заранее рассчитанные ряды для графиков.

Сегмент самодостаточен: для чтения не нужен внешний каталог со схемами,
смещениями или словарями. Внешний каталог может ускорять поиск по множеству
сегментов, но не входит в базовый формат.

Основные принципы:

- Один сегмент — один файл. После записи файл не изменяется.
- Формат оптимизирован под S3 и другие объектные хранилища: чтение строится на
  небольшом числе диапазонных запросов.
- Универсальной схемы строк нет. Каждый тип данных хранится в собственной
  типизированной секции и декодируется своим кодом.
- Версия схемы секции входит в `type_id`. Несовместимое изменение раскладки или
  семантики получает новый `type_id`; совместимое расширение Parquet-схемы
  допускается только по правилам реестра.
- Сторонние инструменты вроде DuckDB или pandas не являются целевым API.
  Parquet используется как набор кодировок и метаданных, а не как обещание
  произвольной аналитической совместимости.

## 2. Раскладка файла

```text
segment.pgm
┌─────────────────────────────────────────────┐
│ MAGIC "PGM1" (4 B)                          │
├───────── COLD: снимки ──────────────────────┤
│ встроенные Parquet-секции:                  │
│ pg_stat_activity, pg_stat_statements,       │
│ pg_stat_user_tables, sys.proc.processes ... │
├───────── WARM: словари ─────────────────────┤
│ dict.strings                                │
│ dict.blobs                                  │
├───────── HOT: хвост ────────────────────────┤
│ dict.hot_strings                            │
│ charts                                      │
│ events                                      │
│ концевой каталог                            │
└─────────────────────────────────────────────┘
```

В начале файла лежит магическое число `PGM1`. В самом конце файла лежит
хвостовой указатель с размером концевого каталога. Концевой каталог содержит
таблицу секций: тип, смещение, длину, число записей и контрольную сумму тела
секции.

Целевой размер хвоста `HOT + концевой каталог + хвостовой указатель` — до 1 МБ.
Код чтения обычно читает последний мегабайт одним диапазонным GET-запросом и
получает концевой каталог, события и графики. Если хвост больше, код чтения
дочитывает недостающие диапазоны по смещениям из концевого каталога.

## 3. Концевой каталог

Концевой каталог бинарный. Записи имеют фиксированный размер, чтобы открытие
сегмента не зависело от сериализатора и не требовало лишних аллокаций.

```text
entry: 32 B
  type_id   u32      // тип секции, см. реестр
  flags     u32      // зарезервировано, сейчас 0
  offset    u64      // абсолютное смещение от начала файла
  len       u64      // длина тела секции
  rows      u32      // число строк или записей в секции
  crc32c    u32      // CRC32C тела секции

catalog_meta: 40 B
  min_ts         i64  // минимальная временная метка сегмента, unix usec
  max_ts         i64  // максимальная временная метка сегмента, unix usec
  source_id      u64  // str_id "{cluster_id}/{pg_system_identifier}", 0 = не задан
  entry_count    u32
  format_version u32  // версия контейнера, не версия схем данных
  crc32c         u32  // CRC32C по entries + catalog_meta, поле crc32c = 0
  reserved       u32

tail_index: 8 B
  catalog_len u32     // len(entries) + len(catalog_meta)
  magic      u32      // ASCII "PGM1"
```

Все числа little-endian. `offset` и `len` всегда относятся к телу секции,
без внешних контейнеров S3 или файловой системы.

`source_id` нужен для быстрого чтения хвостов по множеству файлов без опоры на
S3-путь. Значение — `str_id` строки вида `{cluster_id}/{pg_system_identifier}`;
`0` означает, что источник не задан.

Алгоритм чтения:

1. Прочитать последние 8 байт и проверить `magic`.
2. По `catalog_len` найти начало концевого каталога.
3. Прочитать `entries` и `catalog_meta`.
4. Проверить CRC концевого каталога.
5. Читать нужные секции по `offset` и `len`.

Один `type_id` может встречаться в концевом каталоге несколько раз. Это
используется для разбиения больших секций на части. Код чтения должен
обрабатывать такие части в порядке их следования в концевом каталоге.

Для секций класса `10` повтор `type_id` означает разные `entity`, а не части
одной секции. Разбивать секцию графика на части запрещено. Два экземпляра
одного `type_id` графика с одинаковым `entity` — ошибка кода записи.

## 4. Реестр типов

`type_id` выдаётся только через [реестр типов](type-registry.md). Идентификатор
никогда не переиспользуется, даже если тип больше не пишется.

Правила:

- Несовместимое изменение раскладки секции требует нового `type_id`.
- Новая версия схемы старого источника — это новый `type_id`, а не миграция.
- Совместимое добавление необязательной колонки в конец Parquet-схемы допускается без
  нового `type_id`, если это явно разрешено реестром и старый код чтения
  безопасно игнорирует колонку.
- Код чтения молча пропускает неизвестные `type_id`.
- Код чтения может одновременно поддерживать несколько версий одного источника,
  например `pg_stat_activity v4` и `pg_stat_activity v5`.

Реестр может объявлять обязательные секции для профиля источников. Например,
PostgreSQL-профиль требует `reset_metadata` и `instance_metadata` в каждом
сегменте, где есть PostgreSQL-снимки: без них код чтения не может надёжно
интерпретировать reset счётчиков и системные единицы измерения.

Диапазоны:

| Диапазон     | Класс секций |
|--------------|--------------|
| `1_000_000+` | снимки       |
| `2_000_000+` | события      |
| `3_000_000+` | словари      |
| `10_000_000+`| графики      |

Пример: `1_001_001` — `pg_stat_activity v1`,
`10_010_001` — график disk IOPS v1.

## 5. Секции снимков

Секция снимков — самостоятельный Parquet-файл, встроенный в PGM по смещению.
Код чтения открывает его через объект чтения, ограниченный диапазоном секции:
например `io.SectionReader` в Go или `ChunkReader` в `arrow-rs`.

Требования к секциям снимков:

- Сжатие zstd.
- Ключ сортировки фиксируется в реестре рядом с `type_id`.
- Кодировки выбираются по типам колонок, см. раздел 11.
- Группы строк подбираются так, чтобы метаданные не доминировали над данными.
  Базовый ориентир — 30–60 секунд входных данных или сопоставимый объём строк.
- Фильтр Блума включается для `str_id`-колонок, по которым ожидается точечный
  поиск, например по `query_id`.
- Значения по умолчанию не пишутся как отдельные значения. Используются
  необязательные колонки и `NULL`.

Все текстовые значения в секциях снимков заменяются ссылками на словарь.
В самих снимках не хранится SQL, имена таблиц, `cmdline`, `comm`, значения из
`/proc` и другие строки.

Важное ограничение: группа строк не всегда равна временному окну. Для
секций, отсортированных по сущности, min/max по `ts` часто покрывает почти весь
сегмент. Такие секции читаются целиком или крупными диапазонами. Точное
временное отсечение внутри сегмента гарантируется только для секций, где это
явно следует из ключа сортировки.

## 6. Сортировка

Сортировка нужна прежде всего для сжатия. Чем более похожи соседние значения в
колонке, тем лучше работают дельта-кодирование, RLE и словарное кодирование.

Для каждой секции в реестре фиксируются:

- ключ сортировки;
- политика нарезки групп строк;
- причина выбора, если ключ отличается от стандартного для класса данных.

Есть две основные стратегии:

- `entity-first`: сначала идентификатор сущности, затем время. Подходит для
  стабильных счётчиков, где одна сущность наблюдается много раз: запрос,
  таблица, индекс, процесс, cgroup.
- `time-first`: сначала время, затем уточняющий идентификатор. Подходит для
  таймлайнов, volatile-таблиц и данных, которые читаются как последовательность
  состояний.

Конкретные ключи сортировки фиксируются только в
[реестре типов](type-registry.md). Этот документ задает принципы, а не
дублирует таблицу ключей: иначе формат и реестр начинают расходиться.

Если для новой секции нет сильной причины выбирать `time-first`, используется
`entity-first`. Если позже выясняется, что временное отсечение внутри сегмента
важнее сжатия, вводится новый `type_id` с другим ключом сортировки.

## 7. Строки и большие значения

Весь текст сегмента хранится через интернер. Это касается SQL, планов, имен
объектов, `cmdline`, `comm`, строковых значений из `/proc`, полезной нагрузки
событий и имен серий графиков.

```text
str_id = xxh3_64(bytes)
```

Хэш считается по исходным байтам значения, без нормализации кодировки. Для
строк PostgreSQL это UTF-8 bytes, для системных источников — байты после
нормализации коллектора. Пустая строка — обычное значение и получает ненулевой
`str_id`; `NULL` кодируется отсутствием значения или `str_id = 0` в тех полях,
где ноль явно объявлен sentinel-ом.

`str_id` — детерминированное 64-битное беззнаковое значение. В
Parquet-секциях оно хранится строго с логическим типом `UINT_64`: тогда
статистики min/max и заявленный порядок сортировки совпадают с фактическим
беззнаковым порядком данных.

Хранить `str_id` как знаковый `INT64` запрещено: знаковые статистики ломают
отсечение групп строк для хэшей со старшим битом. Если код чтения не
поддерживает `UINT_64`, допустимая альтернатива —
`FIXED_LEN_BYTE_ARRAY(8)` в big-endian. Его байтовый порядок сравнения
совпадает с беззнаковым числовым порядком.

Секции словаря:

- `dict.strings` — короткие строки, например меньше 4 КБ.
- `dict.blobs` — большие значения: планы, длинный SQL, крупная полезная
  нагрузка JSON.
- `dict.hot_strings` — маленький дублирующий словарь для строк, нужных при
  чтении HOT-хвоста: единицы измерения и имена серий графиков, короткие
  подписи событий.

Тип колонки `str` в реестре означает `str_id`, а не конкретную секцию словаря.
По умолчанию код записи помещает значение в `dict.strings` или `dict.blobs` по
порогу размера. Для отдельных колонок реестр может требовать `dict.blobs`
независимо от размера, например для планов запросов.

`dict.strings` хранится как Parquet:

```text
str_id u64
bytes  binary
```

Секция отсортирована по `str_id` и имеет фильтр Блума по `str_id`.

`dict.blobs` использует ту же адресацию:

```text
str_id       u64
stored_bytes binary
full_len     u64
truncated    bool
full_sha256  optional fixed[32]  // только при truncated = true
```

Если значение больше лимита, в сегмент попадает только префикс
`stored_bytes`, выставляется `truncated = true`, а `full_sha256` считается по
исходному значению. Для неусеченных значений `full_sha256` равен `NULL` и не
вычисляется. Стартовый лимит — 1 МБ, окончательное значение нужно подтвердить
на реальных данных.

При открытии сегмента код чтения сразу загружает `dict.strings`: непрерывный
буфер байт плюс индекс `str_id -> (offset, len)`. `dict.blobs` читается по
требованию.
Поверх этого код чтения может держать межсегментный LRU-кэш `str_id -> bytes`.

Обязательный словарный контракт:

- каждый ненулевой `str_id`, на который ссылается любая секция сегмента,
  присутствует в `dict.strings` или `dict.blobs` этого же сегмента;
- один и тот же `str_id` не может одновременно присутствовать в обоих словарях;
- если значение усечено в `dict.blobs`, `str_id` и `full_sha256` считаются по
  полному исходному значению, а не по сохраненному префиксу.

`dict.hot_strings` — не третий источник истины. Это дублирующий
хвостовой кэш: все его записи также присутствуют в `dict.strings`. Код чтения,
которому нужны только `events` и `charts`, сначала использует
`dict.hot_strings`; полный `dict.strings` загружается только при детализации в
снимки или при отсутствии нужной строки в хвостовом кэше.

Строгая часть контракта: все ненулевые `str_id` из заголовков секций графиков
(`unit`, `series_names`, `entity`) и ненулевой `source_id` концевого каталога
присутствуют в `dict.hot_strings`. Для `ref_str` и `payload` событий этот
словарь заполняется по возможности: короткие подписи дублируются, большие
значения читаются через полный словарь.

Коллизии:

- Код записи проверяет коллизии внутри сегмента. Если один `str_id` соответствует
  разным байтам, запись сегмента прерывается и поднимается аварийное событие.
- Значение `str_id = 0` зарезервировано как «значения нет», см. секцию событий.
  Если вход хэшируется в ноль, код записи должен трактовать это как коллизию:
  нулевой id никогда не попадает в словарь как настоящая строка.
- Межсегментный поиск по `str_id` должен подтверждать совпадение побайтовым
  сравнением после загрузки словаря. Коллизия может дать лишнее чтение, но не
  ложный результат.

Стабильный `str_id` ускоряет поиск: чтобы найти SQL-текст за месяц, код чтения
считает `xxh3_64(text)`, проверяет фильтр Блума в словарях сегментов и читает
только кандидатов.

## 8. Секция событий

`events` — полный таймлайн событий внутри сегмента. Это собственный блок, а не
Parquet: данные лежат колонками, тело сжато одним zstd-фреймом и читается одним
проходом.

Запись события:

```text
ts        i64   // delta + varint, unix usec
type      u32   // тип события из отдельного реестра
severity  u8
pid       i32   // 0 = нет PID
ref_str   u64   // str_id связанной сущности, 0 = нет
payload   u64   // str_id полезной нагрузки JSON, 0 = нет
```

События сортируются по `ts`. Полезная нагрузка JSON хранится в словаре, а не
внутри секции событий.

## 9. Секции графиков

Графики рассчитываются при запечатывании сегмента. Один график — одна секция со
своим `type_id`; идентификатор кодирует и смысл графика, и версию раскладки.

Интерфейс обзорных панелей читает только `charts` и не трогает снимки. Если
новый график появился в новой версии кода записи, старые сегменты не
переписываются. Для старых сегментов интерфейс либо не показывает этот график,
либо считает его из снимков медленным путем.

Тело секции:

```text
series_count u8
unit         u64   // str_id, например "ops/s"
entity       u64   // str_id сущности; 0 = уровень инстанса
series_names u64 × series_count
agg_hint     u8    // 0 = gauge, 1 = counter_rate
point_count  u32
ts[]         delta + varint, zstd
values[]     f64 × point_count × series_count
```

Ось времени одна на все серии графика. Значения хранятся по сериям подряд:
сначала все точки первой серии, затем второй и так далее. Перед zstd к `f64`
применяется byte-stream split: отдельно пишутся все байты с индексом 0, затем
все байты с индексом 1, до индекса 7. Это должно быть описано явно в декодере,
а не оставлено как ссылка на Parquet.

Отсутствующее значение кодируется IEEE `NaN`. Интерфейс должен разрывать линию
на таких точках, а не рисовать ноль.

Значения в `values[]` всегда являются готовыми точками: скорости и производные
серии вычислены при запечатывании. Интерфейс не выполняет преобразований.
`agg_hint` описывает семантику точек: `0` — мгновенное значение, `1` — скорость
в единицах в секунду. Он используется только для подписей и правил уменьшения
числа точек. Базовая политика уменьшения числа точек: `gauge` агрегируется
через `last` или `max`, `counter_rate` — через среднее.

Графики не наследуют проблему первой точки серии скорости из секций снимков:
код записи вычисляет их при запечатывании, держа последний шаг предыдущего
сегмента в памяти процесса. Исключение — первый сегмент после рестарта
коллектора: его первая точка серий скорости равна `NaN`.

## 10. Заголовок собственных блоков

Собственные блоки PGM, сейчас это `events` и `charts`, начинаются с общего
заголовка:

```text
block_magic      u32  // ASCII "PGMB"
body_codec       u8   // 0 = raw, 1 = zstd
uncompressed_len u32
body             ...
```

Сжатая длина берется из `len` в записи концевого каталога. CRC в заголовке не
дублируется: CRC32C тела секции уже хранится в концевом каталоге.

`block_magic` нужен по трем причинам:

- быстро обнаруживать ошибку смещения;
- находить собственные блоки при восстановлении поврежденного файла;
- проверять корректность предварительного чтения хвоста.

Цена — 4 байта на блок. Это тот же практический приём, что `PAR1` у Parquet и
магические байты у zstd-фреймов.

## 11. Сжатие

Базовые меры сжатия:

1. Колоночная раскладка везде: Parquet для COLD/WARM, собственные колонки для
   HOT.
2. Сортировка под конкретный тип секции, см. раздел 6.
3. Интернирование строк: в колонках лежит `str_id`, текст хранится один раз.
4. Кодировки по типам данных:
   - временные метки и монотонные счётчики — `DELTA_BINARY_PACKED`;
   - низкокардинальные значения — `RLE_DICTIONARY`;
   - `f64` в графиках — byte-stream split + zstd.
5. Необязательные колонки вместо записи значений по умолчанию.
6. Общая ось `ts` для всех серий одного графика.
7. Усечение слишком больших строковых значений.

Стартовые уровни zstd:

| Данные | Уровень | Обоснование |
|--------|---------|-------------|
| мини-сегменты на горячем пути записи | 3 | быстро, эти данные скоро пересжимаются |
| финальные COLD/WARM-секции | 9-12 | фоновая операция, файл пишется один раз |
| HOT-блоки | 15-19 | блоки маленькие, важен размер хвоста |

Точные уровни не должны быть догмой. Их нужно закреплять после замеров на
выборке реальных сегментов с учётом CPU-бюджета коллектора.

В CI должен быть набор репрезентативных сегментов. Если изменение кода записи
увеличивает суммарный размер сегмента больше заданного порога, это повод
разобраться до слияния изменений.

## 12. Запись и слияние

Формат записи похож на LSM-подход: память -> маленькие готовые части ->
потоковое слияние в финальный сегмент. Отдельного построчного WAL нет. Вместо
него используется журнал мини-сегментов.

Решение зафиксировано осознанно. Классический WAL, то есть построчный лог,
replay и сортировка при запечатывании, отклонён: он требует второй дисковый
формат на каждый `type_id` с почти неизбежным дрейфом относительно
Parquet-схемы, отдельный код восстановления и пиковую сортировку всего сегмента в
памяти. Цена отказа — окно потери при сбое питания растёт с единиц секунд до
интервала сброса мини-сегментов, обычно около 30 секунд. Для метрик это
приемлемо: теряются данные одного хоста, а не всех хостов.

Порядок записи:

```text
шаг сбора:
  источник -> структуры в памяти -> буфер типа

каждые 30 секунд или при переполнении буфера:
  буферы -> сортировка окна -> мини-сегмент
         -> дописать в active.parts
         -> fdatasync(active.parts)

каждые 10 минут или при переполнении журнала:
  active.parts -> потоковое слияние -> segment.pgm.tmp
               -> fsync(segment.pgm.tmp)
               -> rename(segment.pgm.tmp, segment.pgm)
               -> fsync(directory)
               -> загрузка на S3
               -> очистка active.parts
```

`мини-сегмент` — обычный маленький PGM-сегмент с теми же секциями, но без графиков.
Сжатие на этом этапе быстрое, например zstd-3. Словарь мини-сегмента содержит
строки, впервые встретившиеся в этом окне.

Слияние — k-путевое слияние уже отсортированных частей. Память ограничена
головами потоков, например одной группой строк или одним небольшим буфером на
часть, а не полным объёмом финального сегмента. На этапе слияния строятся
графики и пересжимаются финальные секции.

`str_id` стабилен, поэтому словари частей объединяются без ремаппинга ссылок.
Если одна и та же строка встретилась в нескольких мини-сегментах, в финальном
словаре она остаётся один раз.

### Журнал частей

`active.parts` — локальный файл, который только дописывается в конец:

```text
frame_magic u32  // ASCII "PGMP"
part_len    u64
header_crc  u32  // CRC32C frame_magic + part_len
part        ...  // самостоятельный мини-сегмент PGM
```

`part_len` должен быть меньше настроенного лимита кадра. Кадр с длиной за
пределами файла или выше лимита считается повреждённым. Длина части стоит в
начале кадра, поэтому штатный путь читает журнал строго вперёд.
Восстановительный проход может выполнить повторную синхронизацию: искать
следующий `PGMP` в ограниченном окне, проверять `header_crc`, затем
валидировать найденный мини-сегмент PGM через хвостовой указатель, CRC концевого
каталога и CRC секций.

По смещениям в `active.parts` можно читать ещё не запечатанные данные тем же
кодом чтения, что и финальные сегменты. Это даёт запросы почти в реальном
времени без отдельного формата хранения.

### Устойчивость к сбоям

На старте коллектор проверяет локальное состояние:

1. Сбой во время записи кадра. Журнал проходится вперёд; кадр с битым magic,
   header CRC, некорректной длиной или невалидным мини-сегментом PGM считается
   повреждённым. Если повреждённый кадр последний в файле, это обрыв записи:
   файл обрезается перед ним, потеря ограничена буфером памяти и одним
   недописанным мини-сегментом. Если после повреждённого кадра есть валидные
   кадры, найденные повторной синхронизацией, это не обрыв, а порча носителя:
   поднимается аварийное событие, валидные части до и после повреждения
   включаются в слияние, молчаливое усечение середины файла запрещено. Если
   повторная синхронизация не нашла валидного следующего кадра, хвост после
   повреждения карантинится и не участвует в слиянии. Полная валидация
   мини-сегмента PGM выполняется только при восстановительном проходе, а не на
   каждом штатном запуске.
2. Сбой во время слияния. `.tmp` удаляется, `active.parts` остаётся источником
   истины, слияние повторяется.
3. Сбой после `rename`, но до очистки журнала. Если `max_ts` частей не больше
   `max_ts` готового сегмента, части уже вошли в сегмент и журнал можно
   очистить. Операция идемпотентна.

Ожидается обычная семантика локальной POSIX-файловой системы: не NFS,
не s3fs и не FUSE. `.tmp` и финальный сегмент создаются в одной директории.
Финальный `rename` не должен перезаписывать существующий сегмент: используется
`RENAME_NOREPLACE`, `linkat`/`O_EXCL` или эквивалентная проверка. После
создания, усечения, удаления и переименования выполняется `fsync`
соответствующего файла и директории.

### Поведение под нагрузкой

- Буфер типа превысил лимит до планового сброса -> внеочередной сброс.
- `active.parts` превысил лимит размера -> досрочное слияние и сегмент короче
  10 минут.
- Интернер быстро растёт из-за потока уникальных строк -> аварийное событие и
  досрочное слияние, чтобы следующий сегмент начал с пустым интернером.

Короткие части и короткие сегменты допустимы. Их фактический диапазон времени
фиксируется в концевом каталоге.

## 13. Чтение из S3

Обычный путь чтения:

- список сегментов выбирается по ключам S3;
- код чтения делает диапазонный GET последнего мегабайта;
- из хвостового указателя получает размер концевого каталога;
- из концевого каталога получает `events` и `charts`;
- для детализации читает нужные секции снимков и словари.

Дашборд за сутки для 10-минутных сегментов — это 144 хвостовых чтения. Эти
запросы можно выполнять параллельно.

Детализация по снимкам:

- читать секцию целиком или крупными диапазонами;
- отдельно читать `dict.strings`;
- избегать мелких переходов на уровне страниц Parquet по S3.

Мелкие прыжки по страницам Parquet допустимы для локального кэша или локального
файла, но не должны быть базовой стратегией чтения из объектного хранилища.

## 14. Расположение (локально и в S3)

Почасовое партиционирование по времени, вплоть до часа. Одинаково локально и в S3
(в S3 путь — это префикс ключа):

```text
{root}/{cluster_id}/{yyyy}/{mm}/{dd}/{hh}/{first_ts}.pgm
manifest:  {root}/{cluster_id}/{yyyy}/{mm}/{dd}/{hh}/manifest.idx

пример:
pgm/prod-pg14/2026/06/12/14/1749732600000000.pgm
pgm/prod-pg14/2026/06/12/14/manifest.idx
```

Все сегменты одного часа лежат в одной директории/под одним префиксом; имя
сегмента — `first_ts` первого окна (unix usec), лексикографически = хронологически.

Свойства:

- **Час-директория ограничивает число единиц** размером «сколько сеалов за час» —
  ни локальная директория, ни per-hour manifest не растут бесконечно. Это отказ от
  прежней дневной раскладки без подкаталога часа: при многих источниках или частом
  seal по размеру день накапливал бы тысячи файлов в одной директории, что топит
  скан.
- **Дискавери окна = выбор часовых директорий/префиксов, пересекающих `[from,to]`**
  (+ граничная предыдущая: сегмент, начатый в конце часа `H-1` и запечатанный в `H`,
  лежит в `H-1/` по `first_ts`, но данными заходит в `H`; точное пересечение — по
  `min_ts/max_ts`, час-директория — грубый бакет).
- На S3 час-путь — префикс: `ListObjectsV2` по префиксу часа даёт сегменты часа за
  один LIST; `cluster_id` в начале ключа даёт отдельное дерево на кластер (IAM).
  Лексикографический порядок совпадает с хронологическим.
- **Per-hour manifest** (`manifest.idx`) = конкатенация каталог-мет сегментов часа
  (`first_ts/source_id/min_ts/max_ts/section-presence`, не тела) → один GET на час
  вместо GET на каждый сегмент. Оптимизация, не источник истины: нет/битый manifest
  → фолбэк на `ListObjectsV2` + чтение хвостов сегментов. Лимиты S3 на префикс
  (5500 GET/с) на порядки выше нагрузки.

Кто пишет: коллектор пишет сегмент в часовую директорию и аппендит локальный
`manifest.idx` на seal. Выгрузку часовой директории (сегменты + manifest) в S3
делает `pg_kronika-archiver` — S3-путь его зона (`docs/architecture.md`); формат
файлов от локального размещения не зависит.

## 15. Безопасность данных

PGM — не механизм защиты данных. Сегмент почти всегда содержит
чувствительную информацию: SQL-литералы, имена объектов, аргументы процессов,
фрагменты JSON.

Решения на уровне формата:

- `/proc/PID/environ` не собирается и не имеет `type_id`.
- Удаление данных выполняется удалением сегментов целиком. Точечное стирание
  внутри сегмента невозможно из-за иммутабельности.
- Гранулярность удаления — размер сегмента, обычно около 10 минут.

Всё остальное относится к инфраструктуре: TLS, SSE-KMS, IAM/bucket policy,
политики хранения, контроль доступа к локальному кэшу. Формат не делает
полевое шифрование и не маскирует значения внутри сегмента.

## 16. Совместимость

- `format_version` меняется только при изменении контейнера: концевого
  каталога, хвостового указателя, правил размещения секций.
- Эволюция данных происходит через новые `type_id`.
- Код чтения новой версии должен читать старые сегменты, пока поддержка старого
  `type_id` явно не снята.
- Код чтения старой версии должен пропускать неизвестные секции и читать то, что
  умеет декодировать.

## 17. Открытые вопросы

- Порог между `dict.strings` и `dict.blobs`.
- Лимит усечения больших значений.
- Реальные уровни zstd для COLD/WARM/HOT с учётом нагрузки на хост БД.
- Размер предварительного чтения хвоста: 1 МБ — стартовое значение.
- Нужен ли код чтения на Go; если да, какие Parquet-кодировки поддерживаются
  надёжно.
- `byte-stream split` против Gorilla/XOR-кодирования для рядов графиков.
- Интервал сброса мини-сегментов: 30 секунд как стартовая точка, 10 секунд как
  вариант с меньшим окном потери.
- Нужна ли сортировка по временным корзинам для отдельных `entity-first`
  секций, если локальные запросы по временным окнам окажутся горячими.
- Генерация кода из описания типа: структура -> Parquet-схема -> код записи ->
  код чтения -> golden-тесты.
- Несколько инстансов на хосте: сегмент на пару `(хост, кластер)` с
  дублированием ОС-данных в каждом или отдельный поток ОС-сегментов с
  сопоставлением при чтении.
- Поведение при недоступности S3: лимит локального буфера готовых сегментов,
  политика выброса при переполнении, файл блокировки единственного процесса
  записи для директории.
