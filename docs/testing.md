# Тестирование pgKronika

Версия документа: draft-4, 2026-06-12.

Статус: черновик. Документ записан до начала реализации и фиксирует
стратегию тестирования.

Черновое решение по исполнителю: BDD-сценарии хранятся в Gherkin-файлах и
запускаются через `cucumber-rs`. Описания шагов пишутся на Rust,
многострочные блоки SQL и конфигурации передаются через `docstring`, наборы
выбираются тегами: `demo`, `slow`, `full-matrix`. Шаги в сценариях пишутся
по-английски; русские примеры ниже нужны только для пояснения.

## 1. Принципы

Тесты читают сегменты тем же кодом, что и продукт. Исполнитель сценариев —
отдельный крейт в Cargo workspace. Он подключает `kronika-format`,
`kronika-reader` и `kronika-registry` и даёт проверочные функции: открыть
сегмент, проверить формат, прочитать секцию, разрешить `str_id`, посчитать
дельту. Отдельный тестовый декодер не пишется. Слой чтения веб-процесса
проверяется через JSON API; проверки используют тот же код чтения.

Файл может соответствовать формату, но содержать неверные значения. Тесты
проверяют оба утверждения: «файл соответствует формату» и «в файле лежат
ожидаемые значения». Второму посвящён раздел 4.

Сценарий — это документация. Конфигурации, нагрузка и ожидаемый результат
записаны в тексте сценария; сценарий читается как описание поведения системы.

Тесты не ждут реальные интервалы. Всё, что в рабочем режиме происходит по
расписанию, тест запускает явно: интервалы уменьшаются конфигурацией, сброс и
запечатывание вызываются сигналами — см. сигнальный контракт коллектора в
[architecture.md](architecture.md). Завершение операции видно по файловой
системе: сегмент появляется через атомарное переименование, `active.parts`
увеличивается после сброса. У каждого ожидания есть предельное время и
диагностический снимок состояния; бесконечные ожидания запрещены.

## 2. Уровни

```text
unit      парсеры, формат, интернер — внутри крейтов, без PostgreSQL
property  цикл запись-чтение, восстановление, инварианты — proptest
corpus    проверка формата и контроль размера на корпусе сегментов
bdd       живой PostgreSQL + коллектор + проверочный код чтения и JSON API
demo      BDD-сценарий, после нагрузки окружение остаётся работать
```

### Unit-тесты

Табличные тесты на файлах-образцах, без живого PostgreSQL:

- парсеры `/proc` и `/sys` — снимки реальных файлов с разных ядер; поля,
  отсутствующие на старых ядрах, дают `NULL`;
- парсеры логов — stderr и csvlog, обе локали, строки продолжения;
- формат — цикл запись-чтение секции, концевой каталог, CRC, кадры
  `active.parts`;
- интернер — стабильность `str_id`, нулевой хэш обрабатывается как коллизия;
- golden-тесты `kronika-derive`: структура типа → схема → кодек → декодер;
- байтовые эталоны: минимальный сегмент и кадр `PGMP`, посчитанные вручную по
  [segment-format.md](segment-format.md) и сохранённые как фикстуры. Цикл
  запись-чтение не замечает согласованный дрейф кода записи и чтения от
  спецификации, а байтовый эталон замечает.

Правила с длинным горизонтом — «сегмент + 1 ч» у `pg_settings`, возраст
старейшей мини-части, срок хранения сегментов — тоже проверяются на этом
уровне: часы в `kronika-writer` и планировщике сбора передаются как
зависимость, а тест двигает их вручную. BDD-тесты время не подделывают: им
хватает коротких интервалов и сигналов.

### Property-тесты

Минимальный набор инвариантов:

- произвольный набор строк любых типов, записанный в сегмент, читается
  обратно без потерь;
- журнал `active.parts`, обрезанный на произвольном смещении,
  восстанавливается по правилам [segment-format.md](segment-format.md);
  порча в середине файла не приводит к молчаливому усечению;
- несколько мини-частей, слитых в сегмент, дают те же данные, что прямая
  запись того же входа;
- словарный контракт: каждый ненулевой `str_id` разрешим, ни один не лежит в
  двух словарях сразу, контракт `dict.hot_strings` выполняется;
- сортировка секций соответствует ключу из реестра.

### Корпус

Корпус репрезентативных сегментов генерируется в CI на каждом прогоне, а не
хранится: его производит отдельный набор BDD-сценариев с детерминированными
профилями нагрузки.

Контроль размера — сравнение двух генераций в одном прогоне: корпус
генерируется кодом из main и кодом из PR на одном профиле, разница размеров
по секциям сравнивается с порогом. Это рабочий случай `dump diff` из
[architecture.md](architecture.md). Превышение порога — повод для разбора до
слияния изменений.

Совместимость со старыми сегментами тоже проверяется генерацией: исполняемый
файл предыдущего релиза пишет сегменты, новый код чтения должен их прочитать.

## 3. BDD-сценарии

Сценарий собирается из шагов четырёх видов:

```text
дано    PostgreSQL версии X с конфигурацией <...>
        запущен коллектор с конфигурацией <...>
когда   нагрузка: SQL, pgbench, рестарты, порча файлов
        сигнал коллектору: сброс или запечатывание
тогда   утверждение проверочной функции о содержимом сегмента
        утверждение о ответе JSON API веб-процесса
        в таймлайне есть событие <тип>
```

Целевой вид:

```gherkin
Сценарий: reset pg_stat_statements виден как подтверждённый разрыв ряда
  Дано PostgreSQL 16 с конфигурацией:
    """
    shared_preload_libraries = 'pg_stat_statements'
    compute_query_id = on
    """
  И запущен коллектор с конфигурацией:
    """
    collect_interval = 1s
    flush_interval   = 2s
    """
  Когда выполняется pgbench 5 секунд
  И выполняется SQL:
    """
    SELECT pg_stat_statements_reset();
    """
  И выполняется pgbench 5 секунд
  И сегмент запечатан по SIGUSR2
  Тогда сегмент проходит проверку формата
  И в секции 1_020_001 увеличился pg_stat_statements_reset_at
  И в таймлайне есть событие "stats_reset"
  И JSON API отдаёт ряд calls с разрывом на границе reset
```

Последние два «тогда» проверяют разные уровни. Проверка секции показывает, что
метаданные reset записаны. JSON API показывает, что `kronika-diff` разорвал
ряд по этим метаданным.

Группы сценариев в порядке приоритета:

1. Штатный сбор по каждому источнику: секция есть в сегменте, baseline и
   coverage записаны по контракту.
2. Деградации: расширение не установлено, ошибка прав доступа
   (`permission denied`), источник недоступен — `NULL` вместо нулей,
   диагностическое событие, увеличение соответствующих служебных метрик на
   `/metrics`, метрика-пульс не отстаёт.
3. Reset и рестарты: `pg_stat_reset()`, рестарт PostgreSQL, рестарт
   коллектора — правила `reset_metadata` соблюдены.
4. Сбои: kill -9 коллектора на каждой фазе записи, порча `active.parts`
   остановленного коллектора, повторный старт — инварианты восстановления из
   [segment-format.md](segment-format.md).
5. Версии PostgreSQL: колонки, зависящие от версии PostgreSQL, дают `NULL`, а
   не ошибку.
6. Логи: ротация, `copytruncate`, обе локали, перегрузка читателя логов —
   строки отбрасываются с диагностическим событием и счётчиком, по контракту
   из реестра.
7. Архиватор: загрузка, маркер `.uploaded`, недоступный S3. Удаление
   сегментов выполняет коллектор, поэтому сценарий проверяет файловый контракт:
   архиватор пишет маркер, а коллектор учитывает его при удалении.
8. Веб-процесс: файловый режим над сегментами сценария, чтение через
   `DaemonHttp` и из S3-хранилища (MinIO), незнакомый `type_id` показывается
   как «требуется более новая версия» — по фикстурному сегменту с
  синтетическим будущим `type_id`, потому что внутри одной сборки Cargo workspace
   такой сегмент не получить.

Деморежим — тот же сценарий с другим завершением: после шагов «когда»
автоматические проверки не останавливают окружение. PostgreSQL, коллектор и
веб-процесс продолжают работать, адрес веб-интерфейса печатается.
Демонстрация воспроизводима, а окружение остаётся для ручной проверки.

Деморежим работает без подмен: реальные `/proc` и `/sys/fs/cgroup` хоста,
реальная нагрузка. Фикстурные деревья — инструмент повторяемых тестов, где
вход контролируется и результат сравним от прогона к прогону; в demo им
делать нечего.

## 4. Проверка данных

Живой PostgreSQL и ОС дают фоновый шум, поэтому сценарий сам создаёт факты с
уникальными метками и ищет их отражение в сегменте:

```text
CREATE TABLE bdd_7f3a + 1000 INSERT
  → 1_013_001: строка relname = "bdd_7f3a", дельта n_tup_ins равна 1000

SELECT pg_sleep(2) /* bdd-slow-7f3a */ при log_min_duration_statement = 100
  → 1_026_001: sql содержит метку, duration_ms ≈ 2000
  → 1_001_001: есть снимок, где этот запрос в state = active

пять ошибок 'bdd-err-7f3a' в одном окне агрегации
  → 1_022_001: `pattern` содержит метку, count = 5

два соединения с конфликтом lock
  → 1_011_001: дерево с корректными root_pid и depth
  → события lock_wait_begin / lock_wait_end в таймлайне
```

Метки заодно проверяют словарную цепочку: значение должно разрешаться из
словаря сегмента.

Утверждения о значениях бывают трёх видов:

- точные: дельты C-счётчиков при контролируемой нагрузке (вставили 1000
  строк — дельта равна 1000), число строк, наличие и отсутствие секций,
  `is_baseline`, `NULL` там, где версия PostgreSQL не даёт колонку;
- с допуском: скорости, тайминги, точки графиков — например, TPS в пределах
  ±20 % от показанного pgbench;
- инварианты: каждый `ts` лежит внутри `[min_ts, max_ts]` сегмента,
  C-счётчики монотонны внутри сегмента кроме подтверждённого reset, `NaN`
  стоит ровно на первой точке серий скорости после рестарта коллектора.

Перекрёстная проверка: один факт должен сходиться разными путями.

```text
дельта xact_commit (1_005_001) ≈ интеграл графика TPS (10_002_001)
                               ≈ отчёт pgbench
checkpoint виден в счётчиках 1_006_001, в лог-типе 1_024_001
                               и в графике 10_001_001
процесс с известным comm крутит busy-loop 3 секунды
  → дельта utime / hz ≈ 3 в 1_100_001
запись 100 МБ с fsync
  → дельта write_sectors × 512 ≥ 100 МБ в 1_108_001
```

Дельты, допуски и предикаты считают проверочные функции исполнителя; при
контролируемой нагрузке это вычитание двух точек. Reset-правила и
восстановление baseline не дублируются в проверочном коде: это зона
`kronika-diff`. Она проверяется собственными unit-тестами, тестами на
случайных данных и через
JSON API веб-процесса, который использует `kronika-diff` в обычном пути чтения.

## 5. Шаги нагрузки

Нагрузка — обычные шаги сценария, а не внешние скрипты. Их выполняет
исполнитель сценариев: отдельный крейт в Cargo workspace, в релизный архив не
входит. Стартовый словарь шагов:

```text
выполнить SQL: <...>
выполнить SQL N раз [в M соединениях]: <...>
pgbench N секунд [с профилем <...>]
открыть соединение "имя" и выполнить: <...>     соединение остаётся открытым
завершить соединение "имя"
рестарт postgresql [immediate]
послать коллектору SIGUSR1 | SIGUSR2 | SIGTERM | kill -9
испортить active.parts: обрезать или переписать байты по смещению
                        (только при остановленном коллекторе)
переписать файл фикстурного дерева /proc | cgroup: <...>
ждать события <тип> | появления сегмента
```

Именованные соединения дают сценарии с конкуренцией: конфликт lock — это два
соединения и три шага. Порча файлов — часть словаря, чтобы сценарии сбоев
писались тем же языком, что и штатные сценарии. Словарь расширяется только
вместе со сценарием, которому новый шаг нужен.

## 6. Требования к инструментам

- для тестов нужны малые интервалы коллектора: базовый шаг сбора, сброс
  мини-части, запечатывание. В рабочей конфигурации нижние пределы могут быть
  жёстче;
- сигнальный контракт коллектора — описан в
  [architecture.md](architecture.md);
- проверка формата и чтение секций оформлены как функции крейтов чтения,
  доступные исполнителю сценариев напрямую, без подпроцессов;
- корни `/proc` и `/sys/fs/cgroup` — настройки коллектора. Сценарий
  подкладывает фикстурное дерево и меняет его файлы шагами: переписали
  `memory.events` с выросшим `oom_kill` — коллектор должен выдать событие.
  Контракт коллектора — прочитать и нормализовать; поведение ядра не
  тестируется. cgroup v1 и v2 — два набора фикстур;
- веб-процесс открывает файловый режим над директорией сценария без
  конфигурации.

## 7. Окружение

Интеграционное окружение — один Nix-образ со всеми исполняемыми файлами:

- все поддерживаемые основные версии PostgreSQL: initdb, postgres, psql,
  pgbench; выбор версии — путь к исполняемым файлам;
- расширения: pg_stat_statements для каждой версии, оба варианта
  pg_store_plans;
- исполняемые файлы проекта;
- MinIO или аналог для S3-сценариев.

PostgreSQL поднимается через initdb во временной директории, без Docker. Один
образ фиксирует всю матрицу версий и делает прогон одинаковым на машине
разработчика и в CI.

cgroup-сценарии (`1_200_001+`, OOM, throttling) специального окружения не
требуют: коллектор читает фикстурное дерево через настраиваемый корень,
см. раздел 6.

## 8. Матрица и CI

Оси матрицы:

```text
версия PostgreSQL   все поддерживаемые основные версии: NULL-поведение колонок,
                    pg_stat_checkpointer на PG17+, pg_stat_io на PG16+
локаль логов        en, ru
pg_store_plans      ossc, vadv-форк
роль                primary, standby
cgroup              v1, v2 — два набора фикстурных деревьев
```

Полная матрица не запускается на каждый PR:

```text
на PR   fmt, clippy, unit, property с малым бюджетом итераций,
        корпус, bdd на одной опорной версии PostgreSQL,
        проверка зависимостей крейтов, registry-linter
ночью   полная матрица bdd, property с большим бюджетом,
        контроль размера на полном корпусе
```
