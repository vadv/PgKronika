# Архитектура pgKronika

Версия документа: draft-1, 2026-06-12.

Статус: черновик. Документ добавлен в проект как архитектурное предложение, но
спорные решения пока не переносятся в [формат сегмента](segment-format.md) и
[реестр типов](type-registry.md). Перед реализацией нужно отдельно согласовать:

- коллектор без S3-кода и отдельный `pg_kronika-archiver`;
- один экземпляр PostgreSQL на один процесс коллектора;
- топологии с центральным веб-процессом без S3 и с S3;
- конкретный контракт между локальным хранилищем сегментов, архиватором и
  веб-процессом.

Формат хранения описан в [segment-format.md](segment-format.md), типы данных —
в [реестре](type-registry.md). Этот документ описывает процессы, топологии
развёртывания, границы ответственности, структуру Cargo workspace и правила
версионирования продукта.

## 1. Состав

Проект собирается в четыре статических исполняемых файла под musl. Каждый
процесс запускается отдельно; общей динамической линковки и общей среды
выполнения нет.

```text
pg_kronika-collector   собирает и пишет данные; единственный процесс с повышенными правами
pg_kronika-web         читает данные для людей и агентов: веб-интерфейс, MCP, JSON API
pg_kronika-archiver    переносит готовые сегменты в S3, если это включено
pg_kronika-dump        командная диагностика формата для разработки и поддержки
```

Разделение ответственности:

- данные наружу отдаёт только `pg_kronika-web`;
- живость процесса и собственные метрики отдаёт только `pg_kronika-collector`;
- облачные ключи доступа находятся только в `pg_kronika-archiver`;
- коллектор не содержит S3-кода: S3 не входит в стандартный путь записи.

## 2. `pg_kronika-collector`

```text
                 хост БД
┌──────────────────────────────────────────────┐
│ pg_kronika-collector                         │
│                                              │
│  source-pg   source-os   source-cgroup  logs │
│      └────────┬──────────────┬──────────┘    │
│               ▼              │               │
│   буферы + интернер + предыдущие снимки      │
│               │ запись буфера                │
│               ▼                              │
│        active.parts ──слияние──▶ *.pgm       │
│                                  │           │
│                                  ▼           │
│                       /var/lib/pg_kronika/   │
│                       локальные сегменты     │
│                                              │
│  :9187  /metrics + /healthz — единственный   │
│         слушающий сокет, только метрики      │
└──────────────────────────────────────────────┘
```

Локальный диск — первичное хранилище коллектора. Хранилище сегментов ограничено
по размеру и сроку хранения: что наступит раньше. Стартовая оценка: около
3 ГБ в сутки на хост, неделя локальной истории — около 20 ГБ.

Черновое решение: один процесс коллектора обслуживает ровно один экземпляр
PostgreSQL. Сегментный поток один, ОС-данные пишутся в него же. Несколько
экземпляров PostgreSQL на одном хосте в этой версии архитектуры не
поддерживаются и при проектировании не учитываются. Это решение конфликтует с
открытым вопросом в `segment-format.md` и должно быть подтверждено отдельно.

`/metrics` отдаёт:

- собственные метрики коллектора из `type-registry/semantics.md`, включая
  обязательную метрику-пульс `pgkronika_last_collection_timestamp_seconds`. Она ловит
  состояние «процесс жив, сбор остановился», которое не видно по `up`;
- доменные агрегаты уровня графиков: последнюю точку тех же формул
  `kronika-charts`, которые пишутся в сегменты. Высококардинальные данные по
  запросам и таблицам не попадают в Prometheus-метрики; для них существуют
  сегменты.

Prometheus обычно читает метрики напрямую с коллектора. В такой схеме
`up{job="pg_kronika"}` означает живость процесса сборщика. Для сред, где
коллектору нельзя открывать порт, остаётся режим `--metrics textfile`:
атомарная запись `metrics.prom` через временный файл и переименование для
textfile-модуля node_exporter. Живость в этом режиме проверяется через
метрику-пульс.

Права доступа: чтение `/proc/PID/io` для процессов другого uid может требовать
`setfsuid` через `CAP_SETUID` либо запуск от root. В поставке нужен
unit-файл systemd с минимальным набором прав (`capabilities`).

Сигнальный контракт. Управляющих сокетов у коллектора нет; ручка управления —
сигналы, наблюдаемость результата — файловая система:

```text
SIGUSR1      сбросить буферы в mini-part + fdatasync, работать дальше
SIGUSR2      досрочно запечатать сегмент: сброс буферов + слияние + переименование
SIGTERM/INT  мягкое завершение: сброс буферов → запечатывание → exit 0
```

Мягкое завершение заканчивается запечатыванием: рестарт начинается с пустого
журнала, а данные сразу видны web и archiver. Запечатывание по сигналу и при
завершении генерирует событие `segment_early_seal` с причиной. Этот же
контракт — основа ускорения интеграционных тестов, см. [testing.md](testing.md).

## 3. `pg_kronika-web`

Один процесс открывает три входа к общему ядру чтения:

```text
┌──────────────── pg_kronika-web ─────────────────┐
│  ┌──────────────┐  ┌───────────┐  ┌──────────┐  │
│  │ веб-интерфейс│  │MCP-сервер │  │ JSON API │  │
│  └──────┬───────┘  └─────┬─────┘  └────┬─────┘  │
│         └──────────┬─────┴─────────────┘        │
│              слой аутентификации               │
│                    ▼                            │
│                ядро чтения                      │
│   каталог сегментов, чтение хвостов,            │
│   LRU словарей и str_id, детализация, сравнение │
│                    ▼                            │
│   SegmentStore: LocalDir | FileSet |            │
│                 DaemonHttp | S3                 │
│                                                 │
│  :8688  + API диапазонного чтения сегментов     │
│         из локального хранилища                 │
└─────────────────────────────────────────────────┘
```

- Веб-интерфейс работает через собственный JSON API. Поэтому любая возможность
  интерфейса автоматически доступна MCP-агенту.
- В режиме ожидания процесс не держит ничего, кроме сокета. Кэши словарей и
  сегментов имеют лимит и выгружаются после простоя.
- Файловый режим: `pg_kronika-web ./segment.pgm` или `./dump-dir/` открывает
  веб-интерфейс над скачанными сегментами (`FileSet`). Базовый сценарий
  разбора инцидента: скопировать сегменты к себе и открыть локально.
- Аутентификация не изобретается заново:
  - встроенный basic auth для простых установок;
  - для SSO `pg_kronika-web` ставят за внешний прокси аутентификации, например
    `oauth2-proxy`; сам процесс SSO-протоколов не реализует;
  - для MCP пользователю выдаётся статический bearer-токен:
    `token = HMAC(salt, user)`, где `salt` известна только `pg_kronika-web`, а `user`
    приходит из SSO/basic. `pg_kronika-web` ничего не хранит и проверяет токен
    повторным вычислением. Токен привязан к личности, поэтому для аудита не
    нужна отдельная база. Цена схемы: смена соли отзывает все токены разом,
    поштучного отзыва нет.
- SQL-тексты внутри сегментов считаются чувствительными данными. Анонимного
  доступа нет ни через веб-интерфейс, ни через MCP, ни через JSON API.
- Старая идея похостовой MCP-кластеризации с публикуемым URL и `node_self_id`
  убирается из этой архитектуры. Центральный MCP получает данные по всем
  хостам через хранилища, а `cluster_id` становится параметром инструмента.

## 4. `pg_kronika-archiver`

`pg_kronika-archiver` нужен только там, где сегменты уходят в S3. Он отделён
от коллектора по соображениям безопасности: S3-ключи не должны находиться в
привилегированном процессе, а S3 SDK не должен попадать в его исполняемый файл.

Координация с коллектором идёт только через файловую систему, без IPC:

```text
коллектор: единственный процесс, который удаляет локальные сегменты
архиватор: следит за появлением *.pgm
           переименование атомарно, значит запечатывание завершено
           → проверка CRC контейнера
           → загрузка
           → файл-маркер segment.pgm.uploaded
хранилище: при превышении лимита удаляются сначала сегменты с .uploaded
           сегменты без маркера удаляются только с предупреждением
S3 недоступен:
           локальное хранилище заполняется до лимита → предупреждение
           → дальше по конфигурации: сократить локальную историю
           или остановить удаление
```

`pg_kronika-archiver` не знает о содержимом сегмента: он не зависит от реестра
типов и проверяет только контейнер. Поэтому он обновляется реже остальных
компонентов.

Статус решения: черновик. Эта схема конфликтует со старым конвейером в
`segment-format.md`, где загрузка в S3 пока находится внутри процесса записи
сегмента. До реализации нужно выбрать одну модель.

## 5. `pg_kronika-dump`

`pg_kronika-dump` — командная диагностика формата. Это не просмотрщик для
обычной работы; интерактивный просмотр остаётся в `pg_kronika-web`.
Назначение `dump` — разработка, поддержка и разбор повреждённых или
подозрительных сегментов.

```text
info     концевой каталог: типы, смещения, длины, rows, crc, части, source_id
du       размер секций и Parquet-колонок, коэффициенты сжатия,
         статистика словарей; --json используется в CI-проверке размеров
cat      декодирование секции в stdout: jsonl/csv, --resolve-str
verify   проверка формата: CRC, словарный контракт, покрытие hot_strings,
         порядок сортировки, baseline у changed-типов; exit != 0 при ошибке
parts    кадры active.parts, пробное восстановление
extract  секция как отдельный Parquet-файл для DuckDB/pandas при отладке
diff     разница размеров между сегментами разных версий кода записи
registry известные сборке типы, проверка реестра
```

`pg_kronika-dump` развивается параллельно с кодом записи. Он первый потребитель
крейта для чтения: `verify` — одновременно golden-тест и протокол
поддержки, а `du --json` — инструмент CI-проверки размера на корпусе
реальных данных.

Интерактивного терминального интерфейса не будет: он дорог в разработке и
поддержке. Интерактивный просмотр остаётся в `pg_kronika-web`, включая файловый
режим. Командная строка ограничена табличными командами диагностики.

## 6. Топологии

```text
A. минимум на хосте:        collector (+ node_exporter textfile)
B. хост + web:              collector + web в режиме ожидания
C. несколько хостов без S3: центральный web
                            → API диапазонного чтения веб-процессов на хостах
D. несколько хостов с S3:   C + S3-слой; archiver на хостах;
                            локальное хранилище = свежий кэш, S3 = история
E. офлайн-разбор:           pg_kronika-web ./files/ или pg_kronika-dump
```

Ядро чтения проходит по списку хранилищ с временным разделением: свежие данные
из `LocalDir` или `DaemonHttp`, старые данные — из S3, если он настроен.
Отдельного режима почти реального времени нет: `active.parts` и свежие
сегменты видны через те же хранилища, потому что формат частей самодостаточен.

Хвостовая раскладка сегмента проектировалась под диапазонное чтение как общий
принцип. HTTP Range GET у `DaemonHttp` и S3 GET Range обслуживаются одним и тем
же кодом чтения.

## 7. Структура Cargo Workspace

```text
crates/
├── kronika-format      контейнер: концевой каталог, HOT-блоки,
│                       кадры active.parts, словари, str_id, crc.
│                       Без Parquet и без IO
├── kronika-derive      генерация кода: структура типа → Parquet-схема,
│                       кодировщик/декодер и golden-тест
├── kronika-registry    type_id, схемы, ключи сортировки, классы колонок,
│                       семантики; декодеры всех версий
├── kronika-writer      интернер, буферы, журнал частей,
│                       слияние и запечатывание
├── kronika-store       trait SegmentStore + LocalDir + FileSet
├── kronika-store-http  DaemonHttp-клиент диапазонного чтения
├── kronika-store-s3    object_store + rustls
├── kronika-reader      каталог сегментов, чтение хвостов, LRU, детализация
├── kronika-diff        скорости, восстановление baseline, reset-правила
├── kronika-charts      формулы класса 10: один код для запечатывания, /metrics
│                       и пересчёта старой истории в web
├── kronika-source-pg   сборщики PostgreSQL
├── kronika-source-os   /proc, /sys, cgroup
└── kronika-source-log  читатель логов + парсеры stderr/csvlog
```

Зависимости исполняемых файлов задаются жёстко и проверяются в CI:

```text
collector → writer, source-*, charts, registry      [запрещён store-s3]
web       → reader, diff, charts, store-*, mcp      [запрещены source-*]
archiver  → store-s3, format                        [запрещён registry]
dump      → format, registry, reader, diff, store
```

Смысл правил: S3-код и ключи доступа не попадают в привилегированный процесс;
PostgreSQL-клиент не попадает в веб-процесс; архиватор не требует пересборки при
добавлении типов.

## 8. Версионирование

Три независимые оси:

1. Версия продукта — одна на весь Cargo workspace. Релиз — единый архив из четырёх
   исполняемых файлов. Крейты внутренние, на crates.io не публикуются,
   межкрейтовый semver не ведётся: workspace собирается атомарно.
2. `format_version` контейнера меняется редко. Контракт: исполняемый файл версии X
   читает все `format_version`, выпущенные до X, навсегда.
3. Версии типов, то есть `VVV` в `type_id`, накапливаются в реестре. Декодер
   не удаляется без отдельного решения об устаревании.

Матрица совместимости:

```text
web/dump новее сегмента  → полное чтение
web/dump старее сегмента → незнакомые type_id пропускаются по построению;
                           интерфейс должен показать «N секций требуют более
                           новой версии», молчаливая деградация запрещена
archiver                 → совместим со всем, потому что не знает о содержимом
collector ↔ web          → обмен только файлами и API диапазонного чтения;
                           версионируется контракт файловой системы:
                           имена, маркеры, правила удаления
```

Изменение файлового контракта между collector и web считается событием уровня
`format_version`.

## 9. Сборка

- Цели: `x86_64-unknown-linux-musl`, `aarch64-unknown-linux-musl`.
- Аллокатор: mimalloc глобально во всех исполняемых файлах. Musl-malloc заметно
  деградирует интернер, сортировки и Parquet-кодирование; jemalloc на aarch64
  конфликтует с 16K-страницами.
- TLS — rustls.
- DNS — hickory-dns. DNS в musl урезан: без nsswitch и исторически без
  резервного перехода на TCP.
- Статические файлы веб-интерфейса вшиты в исполняемый файл web через rust-embed.
  Поставка вида «один исполняемый файл + папка с JS» не поддерживается.
- MSRV фиксируется в Cargo workspace.

## 10. Открытые вопросы

- Значения по умолчанию для локального хранилища: 20 ГБ / 7 дней — стартовая
  гипотеза, которую нужно проверить на корпусе реальных сегментов.
- Список узлов для топологии C: статический файл конфигурации, DNS SRV или
  инвентарный файл.
- Подтвердить, что S3 остаётся только в `pg_kronika-archiver` и
  `kronika-store-s3`, а коллектор не содержит S3-кода.
- Подтвердить отказ от нескольких экземпляров PostgreSQL на один процесс
  коллектора или вернуть это в архитектуру как поддерживаемый режим.
