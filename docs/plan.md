# План реализации pgKronika

Версия документа: draft-1, 2026-06-12.

Статус: черновик. Документ фиксирует процесс работы и порядок шагов
реализации. Цели: полное покрытие тестами, читаемость кода, готовность к
публикации как открытого проекта.

Принятые решения:

- код живёт в отдельном репозитории
  <https://github.com/vadv/PgKronika>; этот репозиторий остаётся архивом
  проектирования;
- лицензия — MIT;
- язык кода, rustdoc, комментариев и BDD-сценариев — английский; русские
  проектные документы переводятся по мере стабилизации;
- MSRV — текущий stable Rust, без поддержки старых версий;
- CI — GitHub Actions; покрытие — cargo-llvm-cov с отчётом в PR;
- работа ведётся ветками: один MR — один шаг или логический блок шага,
  каждый MR несёт законченную фактическую ценность и проходит цикл из
  раздела 1.

## 1. Процесс

Каждый шаг проходит один и тот же цикл. Цикл важнее скорости.

1. Проектная записка перед кодом: что пишем, какие публичные типы и функции,
   какие тесты докажут корректность, что осознанно откладывается. Объём —
   полстраницы. Обсуждается и утверждается до первой строчки кода.
2. Код пишется порцией в один логический блок — изменения должны читаться за один
   присест. Попутные изменения вне темы блока запрещены.
3. Критерии готовности: тесты из проектной записки проходят, clippy без
   предупреждений, публичные элементы документированы, у каждого
   нетривиального решения — комментарий «почему» со ссылкой на раздел
   проектного документа.
4. Ревью изменений — до коммита. Непонятный код считается дефектом, даже если
   работает: он объясняется или переписывается проще.

Если в ходе шага обнаруживается пробел в проектных документах, сначала
правится документ, потом код. Код не должен опережать документацию.

## 2. Шаги

Логика порядка: формат определён документами на 100 % и не требует
PostgreSQL — с него начинаем и на нём отрабатываем процесс. Вертикальный
срез делается как можно раньше: он доказывает архитектуру целиком и даёт
BDD-основу, на которой дальше проверяется каждый новый тип данных.

### Шаг 1. Каркас

Workspace со всеми крейтами из architecture.md §7 — пустыми. CI: fmt,
clippy, test, проверка правил зависимостей крейтов (запреты вида
«collector не зависит от store-s3» проверяются автоматически). MSRV, лицензия,
README. Ни строчки доменного кода.

Выход: зелёный CI на пустом workspace; нарушение правила зависимостей
ломает сборку.

### Шаг 2. Ядро kronika-format

Хвостовой указатель, концевой каталог, CRC32C — по segment-format.md §3.

Тесты: байтовый эталон минимального сегмента, посчитанный вручную;
цикл запись-чтение каталога на случайных данных; повреждение любого байта каталога
ловится CRC.

На этом шаге отрабатывается процесс из раздела 1: маленький, полностью
специфицированный кусок.

### Шаг 3. Словари и интернер

`str_id = xxh3_64`, `dict.strings`, `dict.blobs`, `dict.hot_strings`,
усечение больших значений — segment-format.md §7.

Тесты: словарный контракт как инварианты на случайных данных (каждый ненулевой
`str_id` разрешим, ни один не в двух словарях, hot_strings ⊆ strings),
нулевой хэш — коллизия, усечение сохраняет `full_sha256` исходного
значения.

### Шаг 4. Журнал active.parts

Кадры `PGMP`, дозапись, восстановление — segment-format.md §12.

Тесты: проверка с обрезкой на произвольном смещении, порча в середине файла,
повторная синхронизация; молчаливое усечение середины невозможно.

### Шаг 5. Registry и первый тип вручную

`kronika-registry` с минимальной инфраструктурой type_id и один простой тип
с руками написанным кодеком: `1_006_001` (синглтон bgwriter+checkpointer).
Без derive: генерацию пишем после того, как на двух-трёх ручных типах
станет видно, что именно генерить.

Тесты: golden-кодек, registry-linter в CI.

### Шаг 6. Вертикальный срез

Минимальный коллектор: один источник (`1_006_001`), сигнальный контракт,
мини-сегмент → слияние → запечатанный сегмент. Минимальный крейт чтения. Каркас
`cucumber-rs` с базовыми шагами: поднять PostgreSQL из Nix-окружения,
запустить коллектор, послать сигнал, проверить секцию.

Первый BDD-сценарий: поднимаем PostgreSQL → коллектор → SIGUSR2 → в
сегменте есть секция `1_006_001` с правдоподобными значениями.

Выход: сквозной путь работает, тестовая основа готова. Это граница шага 7:
дальше можно параллелить, раньше нельзя.

### Шаг 7. Расширение

Источник за источником, каждый по циклу из раздела 1 и каждый со своим
BDD-сценарием штатного сбора. Параллельно, по мере готовности зависимостей:
`kronika-derive` (после 2-3 ручных типов), charts, diff, web, archiver,
деморежим, набор тестовых сегментов и контроль размера.

Порядок типов внутри шага 7 — отдельное решение по ходу; стартовое
предложение: `1_001_001` activity (горячий путь и словари под нагрузкой),
затем `1_002_001` statements (changed-семантика, baseline, coverage), затем
ОС-снимки.
