# kronika-diff: отложенные работы (v3)

Дата: 2026-07-14. База — followups после мержа PR #62 (`63c85f7`), дополнено
анализом полноты (v2, владелец: F11-F14) и вторым проходом (F15-F16 плюс аудит
diff_key). Каждый пункт берётся отдельным MR.

## Анализ полноты

- **v1 (F1-F10):** оба блокера первого ревью (Б1→F1, Б2→F2), замечания
  архитектора (F5) и перф-панели (F6-F8), разреженные семантики (F3), левый край
  окна (F4), BDD (F9), NaN-унификация (F10).
- **v2 (+F11-F14):** нашёл выпавшие утверждённые решения — per-domain reset-сигнал
  (F11, крупнейшая дыра детекции), честная агрегация (F12), `dealloc` (F13), NULL
  в ключе (F14). Приоритет по цене ошибки в данных.
- **v3 (+F15-F17):** аудит fallback `diff_key` проведён — корректен по всем
  секциям (изначально проверен лишь на трёх; полный проход показал: `scope` —
  per-source константа, ложный флаг; прочие Label вне ключа — атрибуты). Тихой
  порчи схлопывания в смерженном коде нет; остаток — 1-строчная сверка
  `pg_store_plans_vadv`. Плюс wraparound OS-счётчиков (F16, у bigint PG нет) и
  бакетирование дифф-точек для charts (F17).

Пробелов сверх F1-F17 при этом проходе не найдено.

## Честность и корректность

### F1. `NotCollected` + `gated_by` (GUC-гейтинг) — СДЕЛАНО
Реализовано вместе с F5: колонка декларирует `gated_by` на `Bool`-колонку
reset_metadata; дифф-слой переписывает пары, измеренные под выключенным GUC,
в `NotCollected`. Неизвестное состояние гейта ничего не переписывает.
`compute_query_id` остался вне гейтинга: это строковый GUC и ключ серии, не
значение — его судьба решается в F14.

### F2. `entry_epoch` + прокси-эпоха `relfilenode` (OID reuse)
Per-row сигнал «запись пересоздана»: `stats_since` (statements), `relfilenode`
(tables/indexes). Закрывает Б2 точно. Риск сейчас принят осознанно. `relfilenode`
— WRITE-path (коллектор, новая версия layout, BDD).

### F3. Различение `Changed` vs `SnapshotFull`
Разреженная серия: `Changed` мостит к последнему виденному; `SnapshotFull` —
отсутствие = сущность исчезла, повторное появление = `FirstPoint`. Свёртка ветвит
по `contract.semantics`.

### F11. reset-сигналы источников — ПОНИЖЕН (решение владельца, 2026-07-15)
Сигналам источников не верить: `stats_reset` покрывает только официальные
сбросы, а счётчик может провернуться без следа (dealloc записей statements,
OID reuse, пересозданный интерфейс). Первичный детектор остаётся `delta<0`;
«сброс с перерастанием» в pull-модели неустраним сигналами и лечится только
каденцией снапшотов. Слой аномалий устойчив к одиночной ложной дельте by
construction (медиана окна). Остаток пункта: подхватить `reset_at`-колонки
как вспомогательный триггер Reset — низкий приоритет, без доверия сигналу.

## Полнота данных

### F4. reach-back к снапшоту перед окном
Первая точка серии сейчас всегда `FirstPoint` — теряется дельта на левом краю.
Reader-хелпер «последний снапшот до T» через границу сегмента; ретеншн съел
предыдущий сегмент → всё ещё `FirstPoint`.

### F12. Агрегация со счётчиками деградации
Решение 5 (v3/v4): агрегат несёт `series_total` / `series_excluded{reason:count}`.
Эндпоинт сейчас отдаёт per-series без агрегации — решение не потеряно, но и не
записано; молчаливый недоучёт въедет вместе с charts.

### F13. `dealloc` — честность хвоста pg_stat_statements
Вытеснение при переполнении `pg_stat_statements.max` — локальный сброс серий без
сигнала. `pg_stat_statements_info.dealloc` — единственный индикатор потерь; не
собирается. WRITE-path (одним заходом с F2); diff по statements отдаёт rate
`dealloc` рядом.

## Типы контракта

### F5. Типизированные ссылки `SectionColumnRef` — СДЕЛАНО
Введён вместе с F1: `SectionColumnRef { section, column }` в реестре,
`lint_references` резолвит ссылки на полной таблице (кодек-тесты линтят
одиночный контракт и ссылок не видят). `entry_epoch`(F2) при реализации
использует тот же тип для колонки своей секции.

### F14. Семантика NULL в компонентах ключа серии
`queryid` nullable и при `compute_query_id=on` (непланируемые statements).
NULL-строки схлопываются в одну серию-корзину — легитимно, но не задокументировано.
Одна фраза в спеке + doc `diff_key` + тест, что NULL-корзина не ломает соседние.

### F15. Аудит fallback `diff_key` — ПРОВЕДЁН, чисто
`diff_key` при пустом `identity` берёт `sort_key \ {ts}` как сущность. Верно для
секций с `sort_key = (сущность…, ts)`; изначально проверено лишь на statements/
os_cpu/синглтоне, остальные приняты на веру. Аудит пройден по каждой
Cumulative-секции без `identity`:

- **Ложный флаг `scope`** (os_cpu/diskstats/netdev/psi/process/cgroup_*): `scope`
  (OsScope) — per-source константа. `net_scope` возвращает один scope (netns
  коллектора), коллектор бежит в одном месте, внутри секции scope не варьируется.
  diff_key его роняет безопасно: в данных всегда один scope.
- **Прочие Label вне ключа** (device, ppid, comm, datname, usename, plan) —
  атрибуты, функционально определяются ключом-сущностью; не идентичность.
- **`pg_store_plans_vadv`** роняет `queryid_stat_statements` из ключа (сосед
  `pg_store_plans_ossc` держит queryid в sort_key). Избыточен: planid — хэш плана,
  план принадлежит одному запросу, так что `(dbid,userid,planid)` идентифицирует
  сущность. Остаётся 1-строчная проверка семантики расширения vadv (гарантирует ли
  оно planid уникальным на запрос); если нет — объявить явный `identity` с queryid.

Вывод: тихой порчи схлопывания в смерженном коде нет. Остаток — только сверка
pg_store_plans_vadv, низкий приоритет.

### F16 (новое, мелкое). Wraparound фикс-ширинных OS-счётчиков
Дифф трактует любое падение как `Reset`. Счётчики PG — bigint (переполнения нет,
зафиксировано). Но часть OS-счётчиков (легаси-32-бит сетевые/диск) может
переполниться → падение → `Reset`, теряется дельта витка. Проверить, есть ли в
секциях реально фикс-ширинные оборачивающиеся счётчики; если да — коррекция по
известной ширине или осознанное «оборот = Reset».

## Производительность

### F6. Хэш-группировка вместо `BTreeMap`
`FxHash` O(rows) вместо `BTreeMap` (детерминизм сейчас); по показаниям бенча (F7),
с сохранением детерминизма выдачи.

### F7. criterion-бенч дифф-стадии
Многосерийный вектор (10000 строк / N уникальных `queryid`) в `serving.rs`. Сейчас
заглушка `use kronika_diff as _`.

### F8. Микро-оптимизации свёртки
Клон `String` в ключ; пред-резолв индексов колонок раз на секцию. По F7.

### F17 (новое, для charts). Бакетирование дифф-точек
Широкое окно × частые снапшоты → тысячи точек на серию; сейчас `DIFF_MAX_ROWS`
жёстко отклоняет. charts потребует rate по временны́м бакетам (down-sample), не
per-pair. Скорее скоуп charts, но упереться в отклонение можно раньше.

## Тесты

### F9. BDD-сценарий на `/v1/diff`
Дельта по двум снапшотам; `Reset` по `pg_stat_reset`; пустой стор. Плюс сценарий
«сброс с перерастанием» (нагрузка после reset поднимает счётчик выше старого —
сейчас пройдёт валидной дельтой; после F11 обязан стать `Reset`) — записать сразу
pending/xfail-маркером честности.

## Чистка

### F10. Единая обработка не-конечных f64 в `serialize.rs`
`value_to_json` паникует на NaN/inf, `finite()` маппит в null — унифицировать.

## Приоритет (по цене ошибки в данных)

1. **F5 → F1 + F11** одним заходом (тип ссылок, co-загрузка reset_metadata, GUC и
   per-domain сигнал): закрывает Б1 и крупнейшую дыру детекции сброса.
2. **F3** (разреженность) и **F4** (левый край) — полнота данных.
3. **F2 + F13** одним WRITE-path заходом (relfilenode + dealloc в сбор).
4. **F9** (BDD, включая xfail «перерастание»), **F12** — с первой агрегацией.
5. **F7 → F6/F8** по бенчу; **F10, F14, F16, F17** — фоновая чистка/по мере нужды.
6. **F15** — аудит проведён (чисто); остаток — только сверка `pg_store_plans_vadv`,
   низкий приоритет.
