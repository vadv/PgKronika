## Принцип

Бенчмарк-управляемо. Оптимизация без замера до/после НЕ коммитится. Baseline и дельты — в леджер в машиночитаемом виде. Каждая задача: бенч → правка → бенч → гейт → SDD-ревью.

## Инварианты (не сломать; сверять на КАЖДОЙ задаче)

- **Форма JSON-ответа** идентична (BDD + golden сверяют точные значения по union-колонкам, gaps, next_cursor).
- **Порядок строк** (sort по sort_key + остальные union-колонки) и **keyset-курсор** идентичны. Курсор = ПОЛНЫЙ union-вектор в порядке `logical.columns`.
- **Stale-семантика**: активный юнит мог уйти под рассинхрон во время чтения → ретрай ≤ MAX_REFRESH через немедленный `refresh()`, потом деградация в gap (НЕ ошибка наружу). Сохранить 1:1 (см. P2).
- **Readiness**: `last_refresh`/health-метрики двигаются на КАЖДЫЙ успешный refresh-тик, иначе idle-стор даст ложный `/readyz` 503 (см. P3).

Факты, подтверждённые ревью (снимают риски):
- Весь декод-путь (`gather`, `open_unit`, `dictionary`, `decode_rows`, `catalog`, `units`, `handles`) УЖЕ `&self`. Единственные `&mut self` на снапшоте — `refresh`/`refresh_incremental`. Значит `&mut` в `section()` нужен ТОЛЬКО ради `snap.refresh()` в retry (section.rs:122). P2 безопасен по владению.
- `Cursor.values: Vec<Value>` УЖЕ позиционный в порядке union-колонок → P1b его выравнивает со строками (убирает переупаковку `cursor_row` section.rs:150-154), а не усложняет.
- BDD НЕ покрывает stale (харнес читает только sealed). Стейл держат 2 юнит-теста ридера (section.rs:812 `active_unit_removed_mid_read_degrades_to_a_gap`, section.rs:1324 `stale_active_unit_refreshes_and_reads_the_new_part`) — их гонять через новую обёртку.

## Задачи

### P0 — Бенч-инфраструктура (честная база)

criterion как dev-dep в `kronika-reader`. `crates/kronika-reader/benches/serving.rs`.

- **mimalloc в бенчах ОБЯЗАТЕЛЕН** (прод/web-bin на mimalloc; без него cargo bench ридера возьмёт glibc и цифры аллокаций #1/#2/#4a не перенесутся). `#[global_allocator] = mimalloc::MiMalloc` в бенче + mimalloc в dev-deps ридера.
- **Фикстура = `pg_stat_statements`** (W≈55, union нескольких версий V1+V4+V6 в одном окне — чтобы Null-fill отсутствующих колонок попал в замер). N∈{100, 10000}.
- **Две фикстуры по sort_key**: уникальные `ts` И массовые НИЧЬИ по `ts` (реалистично для statements: много строк с одним timestamp снапшота). O(W²) tie-break бьёт ТОЛЬКО при ничьих — без них #3 недозамерится.
- **Бенчи**: (1) `sort_only` — изолированное `compare_full`/sort над готовым `Vec<OutRow>` БЕЗ open_unit/I/O (иначе файловый шум размажет сигнал sort); (2) `section_query` — полный gather+sort+пагинация; (3) `page_to_json` над готовой SectionPage; (4) `refresh_incremental` раздельно на unchanged-journal и appended-tail тик.
- **Счётчик аллокаций**: counting global-allocator-обёртка (или бенч в web-крейте) — мерить ЧИСЛО аллокаций для #1/#2/#4a, не только время. criterion сам этого не даёт.
- Прогрев page cache + `black_box`; зафиксировать в леджере, что бенч не с холодного диска.
- Гейт: бенчи компилятся и гоняются; baseline-числа в леджер.

### P1a — Индексное сравнение (лечит #3 O(W²)), БЕЗ смены публичного типа

`OutRow` остаётся `Vec<(String,Value)>`, но собирается в порядке `logical.columns` (уже так, section.rs:243-252). Значит сравнение можно вести по ИНДЕКСУ, а не линейным `row_value`.
- Предвычислить индексы `sort_key`-колонок в `logical.columns` ОДИН раз до `sort_by` (section.rs:137-139).
- `compare_full`/`compare_by_sort_key`: сравнивать по позиции `a[i].1` vs `b[i].1`, убрать `row_value` (линейный `.find`) и `sort_key.contains` в цикле.
- Замер: `sort_only` bench на фикстуре с ничьими. Ожидание: O(N log N · W²) → O(N log N · W), заметно на W≈55.
- Риск НИЗКИЙ: чисто внутренний, публичный тип не тронут, существующие тесты sort/cursor зелёные.

### P1b — Позиционный OutRow (лечит #2 аллокации имён)

- `OutRow = Vec<Value>` (позиционный, порядок `logical.columns`).
- `SectionPage.columns: Arc<[&'static str]>` — имена (уже `&'static str` в `LogicalColumn.name`), строятся раз из `logical.columns`, шарятся на страницу дешёвым `Arc`-клоном. НЕ резолвить `logical_section(name)` в сериализаторе (аллокация union + скрытый инвариант позиция↔реестр).
- `gather` (section.rs:243): `Vec<Value>` без `to_owned` имён; строгий Null-fill РОВНО до `logical.columns.len()` (иначе индексация вылетит) — тест `union_across_versions_fills_missing_column_with_null` (section.rs:613) держать зелёным.
- Курсор: `next_cursor` (section.rs:166-169) = `row.clone()` (уже позиционный); `cursor_row`-переупаковка (150-154) УДАЛЯЕТСЯ. ИНВАРИАНТ: курсор кодирует полный union-вектор в порядке columns → при смене union между страницами позиционный курсор мог бы съехать. ДОБАВИТЬ: тест resume на новом представлении + BDD-сценарий «union изменился между next_cursor» (новая версия сегмента с новой колонкой добавилась между страницами → keyset не пропускает/не дублирует).
- `serialize.rs:row_to_json`: `page.columns.iter().zip(row).map(|(name,val)| (name, value_to_json(val)))` — второй клон имён уходит.
- Замер: `page_to_json` bench (аллокации имён) + `section_query`. Риск: меняет публичные `OutRow`/`SectionPage`; потребитель вне ридера ровно один — `serialize.rs`; тест-литералы reader'а `("ts".to_owned(),…)` массово переписываются (механически).

### P2 — Read-only `section`/`sections` (лечит #1 клон-на-запрос)

- `section(&LocalDirSnapshot,…)` / `sections(&LocalDirSnapshot,…)` — `&self`, чистый gather; `GatherError::Stale` пробрасывается наружу.
- Retry-политика ОСТАЁТСЯ В РИДЕРЕ: `pub fn section_retrying(initial: &LocalDirSnapshot, reopen: impl FnMut() -> io::Result<LocalDirSnapshot>, name, source, from, to, limit, cursor) -> Result<SectionPage, QueryError>` (+ аналог для sections). Логика: gather по `initial` (заимствовано, БЕЗ клона) → на `Stale` цикл `snap = reopen()?; gather(&snap)` ≤ MAX_REFRESH → потом gap. MAX_REFRESH и деградация — в одном месте ридера, под юнит-тестами.
- Веб-хендлеры (v1.rs:146,175): `let base = state.snapshot.load_full();` (Arc, БЕЗ клона) → `section_retrying(&base, || { let mut s = (*base).clone(); s.refresh()?; Ok(s) }, …)`. КЛОН ТОЛЬКО В reopen-замыкании (вызывается лишь на Stale). Happy-path (99.9%) — без клона.
- Reopen делает `clone()+refresh()` (синхронный ре-скан), НЕ `load_full()` повторно (иначе между тиками фоновой таски вернётся ТОТ ЖЕ stale → busy-loop; refresh даёт немедленный прогресс). Так stale-семантика сохраняется 1:1 с текущей.
- Юнит-тесты ридера (section.rs:812, 1324): перевести на `section_retrying` с reopen `|| { snap2.refresh()?; Ok(snap2.clone()) }` — покрытие retry-пути сохранить.
- Замер: `section_query` bench + нагрузочный (concurrent load_full+section vs текущий clone). Риск средний (stale), но снят по владению (декод уже &self). Оба хендлера через ОДИН хелпер (не дублировать MAX_REFRESH).

### P3 — Idle-gate рефреша (лечит #4; наименее приоритетно — коммит по замеру)

- `LocalDirSnapshot::refresh_incremental(&mut self) -> io::Result<bool>` (changed). `changed` детектит появление И удаление И reset: дешёвый `read_dir`-снимок имён sealed (без `read_catalog` тел) + `valid_len` изменился в ЛЮБУЮ сторону (не только «вырос» — иначе ABA-reset до той же длины и retention-удаление пропущены) + новые active-парты.
- `refresh_loop` (main.rs): `last_refresh.store` + health-gauge (`store_warnings`/`store_damages`) + iterations — на КАЖДЫЙ Ok-тик; `snapshot.store(clone)` republish — ТОЛЬКО при `changed==true`.
- Тесты: changed==true при append/seal/reset/delete; changed==false на неизменном; «changed==false двигает last_refresh, НЕ двигает republish»; негативные — reset-до-той-же-длины, удаление sealed → changed==true.
- Замер: `refresh_incremental` unchanged-тик. Если нет измеримого выигрыша — НЕ коммитить (1 Гц, не горячий путь).

### P4a — str_id без лишнего `into_owned` (кандидат, крупнее #2 на text-секциях)

`cell_to_value` на `StrId` делает `String::from_utf8_lossy(bytes).into_owned()` (value.rs:79) — копия на КАЖДУЮ строковую ячейку. На statements (query и др. text-колонки) доминирует над аллокацией имён. `from_utf8_lossy` возвращает `Cow::Borrowed` при валидном UTF-8 — код его насильно копирует. Замерить долю; убрать лишнюю копию (отдать borrowed, где валидно). Если `Value::Str(String)` → `Cow<'static>`/`Bytes` — делать в ОКНЕ P1b (публичный тип уже ломается, не два breaking-change подряд). Коммит по замеру.

### P4b — Стриминг сериализации вместо материализации DOM (кандидат)

`page_to_json` строит целое `serde_json::Value`-дерево (Map на строку), потом axum сериализует повторно. Прямая `serde_json::to_writer` по стриму строк (позиционных после P1b) убирает промежуточное дерево. Потенциально крупнее #2. Замерить `page_to_json → to_string` отдельно; реализовать если выигрыш есть.

P0 → P1a → P1b → P2 → P3, кандидаты P4a (в окне P1b) и P4b (после P1b) — по замеру. P1a до P1b (алгоритм до смены типа); P2 последний из рисковых (на стабилизированном представлении строк). P3 независим (refresh_loop) — можно раньше. Каждая задача: бенч до/после + гейт + SDD-ревью.
