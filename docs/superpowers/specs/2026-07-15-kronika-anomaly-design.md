# kronika-anomaly: проектная записка (v3)

Дата: 2026-07-15. Статус: черновик на утверждение до кода (plan.md §1).
Основание: исследование kronika-anomaly-research-full.md и ответы владельца
(2026-07-15): ретеншн — несколько часов; сезонность не анализируем — ищем
всплески; push-алертинга нет; v2 — первичный запрос UI: «все аномалии за
период по всем метрикам». **v3 — следствия ревью PR 64**: детектор скользит
по дифф-ряду периода, построенному один раз, а не по пер-оконным диффам;
доступ к диффу — библиотечный diff_section, не HTTP-адаптер.

## 1. Что пишем, зачем, где стоит

kronika-anomaly отвечает UI на вопрос «что и когда серьёзно менялось за
период — по всем метрикам»: всплеск дисковой активности, рост времени
выполнения queryid. Вход — точки производных kronika-diff по всем
зарегистрированным секциям источника; выход — эпизоды аномалий (серия,
колонка, интервал, пиковый скор) единым ранжированным списком.

Вычисление только на чтении, по запросу. Ни фоновых процессов, ни
push-алертов, ни персистентного состояния: ретеншн в несколько часов делает
полный пересчёт из сырья дешёвым (batch-чтение sections() открывает каждый
юнит один раз для всех секций; decode ~21 ms на 10 000 строк), а отсутствие
подписок убирает слой подавления дребезга (k-из-w, гистерезис, latch) — он
нужен потоковому алертингу, которого здесь нет.

## 2. Линия ответственности

- **Реестр** — декларации: классы колонок (Cumulative/Gauge) уже есть;
  floor-константы скоринга привязываются к классам (§4), не к именам.
- **kronika-diff** — производные и статусы качества; скоринг потребляет
  только Value-точки.
- **kronika-anomaly** (новый маленький крейт) — чистая математика: скор окна
  и свёртка эпизодов. Ни имён PG-полей, ни знания секций.
- **web** — эндпоинт: batch-сбор всех секций через существующий
  sections()+diff, прогон ядра, JSON.

## 3. Математика

### 3.1 Скор одного окна (ядро)

Modified z-score Иглевича-Хоглина:

```
med_ref = median(ref);  MAD = median(|ref − med_ref|)
sigma   = max(1.4826·MAD, floor)                      (floor: §4)
M       = (median(cur) − med_ref) / sigma
кандидат: |M| > threshold                              (дефолт 3.5)
```

median(cur), не последняя точка — окно должно «серьёзно поменяться», а не
мигнуть; медиана окна — встроенный анти-дребезг.

### 3.2 Сканирование периода (режим UI)

Ключевое решение v3 (из ревью PR 64): дифф-ряд каждой серии строится ОДИН РАЗ
на весь период — один batch-сбор точек всех секций (sections(), один проход
по сегментам), один вызов diff_section на секцию за [from, to]. Скользящие
окна двигаются по готовому ряду производных локально, по индексам:

```
diff_series = diff_section(rows периода)            (один раз на серию)
позиции t_k = from + W, from + W + s, …, to        (шаг s, дефолт W/4)
для каждой позиции:
  cur = Value-точки diff_series в [t_k − W, t_k]    (срез по индексам)
  ref = Value-точки diff_series периода МИНУС окно  (некаузальный референс)
  M(t_k) = скор §3.1
эпизод = максимальная непрерывная группа позиций с |M| > threshold;
атрибуты: {start, end, peak_M, направление, числа объяснения в пике}
```

Что это закрывает (находки ревью PR 64):
- FirstPoint один на серию за период (честное начало данных), а не на каждую
  позицию окна — пер-оконный дифф терял бы дельту на левом краю каждой
  позиции (reach-back F4 остаётся полезным для самого левого края, но
  перестаёт быть блокером);
- HTTP-гард DIFF_MAX_ROWS не на пути — детектор работает в библиотечной
  плоскости (типизированный diff_section, не section_diff_object —
  тот является JSON-адаптером web-слоя);
- декод и дифф не повторяются на перекрывающихся окнах.

Некаузальный ref — осознанное решение: это ретроспективный разбор, не
стриминг; «как окно отличалось от остального периода» — ровно семантика
подсветки, и данных для медианы больше. Робастность медианы/MAD держит
загрязнение ref до 50%: эпизод длиной в час при пяти часах периода — 20%.
Честное ограничение: всплеск, покрывающий больше половины периода, не
детектируется — референс сам аномален; записано, UI-графики это покажут.

### 3.3 Гейты достаточности и качества

- ref ≥ 20 точек и cur ≥ 3 точки на позицию, иначе позиция не оценивается;
  серия целиком без единой оценённой позиции → NotEvaluated { reason }
  (разреженные per-query серии отсеиваются честно, не шумят).
- NoData-точки diff (Reset/Gap/FirstPoint/Anomaly/NotCollected) в окна не
  входят; их количество возвращается в ответе.
- Направление — знак M; всплеск и провал различимы.

## 4. Floor — декларативно, по классам

MAD вырождается в 0 при >50% одинаковых значений ref (тихие счётчики:
deadlocks, temp_files):

```
floor = max(eps_abs(class), eps_rel · |med_ref|)
```

eps_rel — глобальный дефолт 0.05, параметр запроса; eps_abs — константа
класса колонки в реестре (канонического литературного значения нет —
калибруется; дефолты живут в контракте, не в коде скоринга). Следствие у
тихих серий: первый deadlock за часы даёт огромный |M| — желаемое поведение.

## 5. Публичный API

Ядро — две чистые функции:

```
score_window(cur: &[f64], ref_: &[f64], params: &ScoreParams) -> Scored
Scored = Evaluated { m, dir, med_cur, med_ref, mad_ref, sigma_used, n_cur, n_ref }
       | NotEvaluated { reason: RefTooSmall | CurTooSmall | AllNoData }

episodes(profile: &[(ts, Scored)], threshold) -> Vec<Episode>
Episode = { start, end, peak_ts, peak: Evaluated }
```

Обе детерминированы, без I/O; Evaluated — готовое объяснение числами
(«rate 180/s при медиане периода 42/s; 6.2 робастных сигмы, порог 3.5»).

Эндпоинт (первичный, кросс-секционный):

```
GET /v1/anomalies?from=…&to=…&window=60m&step=15m&threshold=3.5&limit=50
```

— batch-скан ВСЕХ зарегистрированных секций источника (sections() читает
каждый юнит один раз), на каждую серию (identity контракта) каждую
Cumulative-колонку — профиль M и эпизоды; Gauge-колонки скорятся тем же
ядром по сырым значениям; Label/Timestamp не скорятся. Ответ: эпизоды,
глобально отранжированные по peak|M| (валюта сравнима между секциями — в
этом смысл MAD-единиц), limit по эпизодам; счётчики честности
{ sections_scanned, series_total, evaluated, not_evaluated{reason},
nodata_points } per section. Фильтр ?section=name — тот же код, суженный
скан (страница секции).

Бюджет производительности: полный скан типового периода — целевой ≤ 1-2 s;
подтверждается criterion-бенчем (§6). Бенчи закладываются ДО реализации
эндпоинта (рекомендация перф-ревью PR 64): батч-сбор против section×N,
diff_section изолированно (хвосты F6 BTreeMap-группировки и F8
String-на-ячейку бьют по многократной свёртке), пик аллокаций по числу
секций.

## 6. Тесты, доказывающие корректность

Юнит score_window (чистая, без моков):
- всплеск ×5 при спокойном ref → |M| > 3.5, dir положительный;
- спокойные окна → |M| < 1;
- ref загрязнён на 20% инцидентом → порог держится;
- MAD = 0 + отклонение → floor работает, M конечен и велик;
- провал → отрицательный M, симметрия;
- гейты ref/cur → NotEvaluated с точной причиной;
- floor: выбирается max(eps_abs, eps_rel·|med|).

Юнит episodes:
- один всплеск → один эпизод, границы совпадают с превышением порога;
- два раздельных всплеска → два эпизода;
- касание порога без превышения → ноль эпизодов;
- пик эпизода — позиция максимального |M|.

Интеграционные (web): фикстура с искусственным всплеском одной серии одной
секции → её эпизод первый в глобальном top-N; NoData исключены и посчитаны;
criterion-бенч полного скана (матрица секций × строк) с фиксацией бюджета.
BDD: спайк нагрузки между сборками → /v1/anomalies подсвечивает затронутые
серии с корректным интервалом.

## 7. Осознанно отложено

- Сезонные baseline — беспредметны при ретеншне в часы.
- Слой подавления (k-из-w, гистерезис, latch), CUSUM-дрейф, BH-гейт —
  атрибуты push-алертинга, которого нет.
- Poisson-полоса для редких счётчиков — floor покрывает грубо; уточнить
  после обкатки.
- log-преобразование таймингов — rate от time-счётчиков это утилизация,
  уже нормирована.
- DDSketch/роллапы — нечего сжимать.
- Каузальный режим ref (история строго до окна) — понадобится только если
  появится push/стриминг.

## 8. Порядок реализации

1. Крейт kronika-anomaly: ScoreParams/Scored/Episode + score_window +
   episodes + юниты §6.
2. eps_abs в реестр (константа класса; линтер: неотрицательность).
3. Эндпоинт /v1/anomalies поверх batch sections()+diff, скользящие окна
   одним проходом; интеграционные тесты + criterion-бенч бюджета.
4. BDD-сценарий со спайком; фиксация дефолтов в доке эндпоинта.
