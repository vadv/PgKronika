# Правила и семантика реестра

## 1. Схема нумерации

```text
type_id = C_SSS_VVV

C   — класс секции:
      1  снимки
      2  события
      3  словари
      10 графики

SSS — источник внутри класса
VVV — версия раскладки, начиная с 001

пример:
1_001_001 — pg_stat_activity, раскладка v1
```

Для класса `1` используются такие диапазоны источников:

| Диапазон `SSS` | Содержимое |
|----------------|------------|
| `001-099` | PostgreSQL |
| `100-199` | `/proc` и `/sys`, кроме cgroup |
| `200-299` | `/sys/fs/cgroup` |
| `300+` | резерв: PgBouncer, Patroni и другие источники |

## 2. Семантика сбора

Семантика сбора — часть контракта типа. Её изменение требует нового
`type_id`, даже если список колонок не поменялся.

### Периодичность сбора

Базовый шаг регулярного сбора — 5 секунд. Конкретный источник может задавать
другой период в реестре: например 30 секунд, 5 минут, потоковый сбор или сбор
по изменению.

Реализация может пропустить регулярный сбор под нагрузкой, но пропуск должен
быть виден коду чтения: через разрыв временных меток, событие `collector_gap`
или служебную метрику коллектора. Молчаливое «сжатие времени» запрещено.

### `snapshot_full`

При каждом регулярном сборе пишутся все строки источника.

Отсутствие сущности в снимке означает, что сущность исчезла: процесс завершился,
backend отключился, cgroup пропала и так далее. Коллектору не нужно хранить
состояние источника, кроме интернера строк.

### `conditional_full`

Источник пишет полный снимок только когда выполнено условие сбора. Если условие
ложно, секция может отсутствовать; это означает пустой результат, а не unknown.

Пример: дерево ожиданий `pg_locks`. Если предварительная проверка не нашла
ожиданий lock, секция отсутствует, и код чтения трактует это как «ожиданий lock
в окне не было».

Условие сбора — часть контракта типа и фиксируется рядом со схемой.

### `event_stream`

Пишутся все события, которые коллектор увидел в интервале сегмента. Это не
снимок состояния. Отсутствие секции означает, что событий этого типа не было
или источник был недоступен; если источник недоступен, коллектор должен
сгенерировать диагностическое событие/метрику, чтобы код чтения не принял это
за тишину.

Пример: логовые типы PostgreSQL.

### `changed`

Используется для тяжёлых top-N источников: statements, plans, таблиц,
индексов. Пишутся только строки, у которых изменились кумулятивные счётчики,
плюс обязательный baseline в начале каждого сегмента.

Состояние коллектора:

```text
prev: Map<EntityKey, Row>
```

`prev` хранит последний полный снимок источника, а не отфильтрованный результат.

Алгоритм:

```text
на каждом сборе:
  rows = полный top-N снимок источника

  если это первый сбор в сегменте:
      писать все строки, is_baseline = true
  иначе если len(rows) < FILTER_MIN:
      писать все строки
  иначе писать строку, если:
      - сущности нет в prev;
      - изменился хотя бы один C-счётчик;
      - изменился hash L-колонок, не входящих в ключ сущности;
      - сработал always_include-хук типа.

  prev = полный снимок rows
```

Стартовое значение `FILTER_MIN` — 500 строк. На меньших объёмах экономия
незначительна, а мерцание состава строк усложняет чтение.

Обязательные правила:

- Baseline пишется один раз в каждый сегмент. Baseline только при старте
  коллектора запрещён: такой сегмент нельзя читать самодостаточно.
- Отсутствие строки не означает исчезновение сущности. Код чтения восстанавливает
  состояние из baseline и последующих изменений.
- Отрицательная дельта C-счётчика означает reset статистики: строка пишется
  принудительно, генерируется событие, а `reset_metadata` подтверждает причину.
- `prev` обновляется полным снимком. Обновление только отфильтрованными
  строками накапливает дрейф фильтра.
- Изменение L-колонок, не входящих в ключ сущности, пишется даже без изменения
  C-счётчиков. Иначе переименование таблицы, изменение `indexdef`, `tablespace` или
  похожий атрибут могут стать невидимыми.
- `always_include` разрешён для признаков класса `G`, которые важны сами по себе:
  резкий рост `n_dead_tup`, `size_bytes`, retained WAL и похожие признаки.
- Для top-N источников реестр должен фиксировать порядок отбора, `max_n`,
  cutoff-метрику и coverage-поведение. Код чтения не должен принимать top-N за
  полный набор данных.

### `on_change`

Источник пишется при изменении или по длинному таймеру. Примеры:
`pg_settings`, `mountinfo`, `cpuinfo`.

Правила:

- значение обязательно пишется при старте коллектора;
- изменение обычно определяется по хэшу нормализованного содержимого;
- для каждого типа фиксируется политика материализации:
  `every_segment_last_known` или `sparse_lookback`;
- `every_segment_last_known` пишет актуальную копию в каждый сегмент, даже если
  источник не изменился;
- `sparse_lookback` разрешает секции отсутствовать в большинстве сегментов, а
  код чтения ищет последнее известное значение в предыдущих сегментах.

Для источников, которые нужны для интерпретации данных текущего сегмента,
используется `every_segment_last_known`. В текущем реестре это `pg_settings`,
`mountinfo` и `cpuinfo/topology`. `sparse_lookback` допустим только для
источников, отсутствие которых не ломает самодостаточное чтение сегмента.

## 3. Классы колонок

Каждая колонка типа аннотируется одним из классов:

| Класс | Значение | Использование |
|-------|----------|---------------|
| `C` | cumulative | монотонный счётчик; для графиков обычно используется дельта за время |
| `G` | gauge | мгновенное значение |
| `L` | label | идентичность или атрибут сущности |
| `T` | timestamp | временная метка, `i64` unix usec |

Базовые типы данных:

```text
i8 i16 i32 i64
u8 u16 u32 u64
f32 f64
bool
str  = str_id u64
ts   = i64 unix usec
тип? = колонка с `NULL`, например i32? или ts?
```

Правила для C-счётчиков:

- отрицательная дельта считается reset;
- reset не сглаживается и не превращается в ноль;
- причина reset должна быть подтверждена служебными типами или событием, когда
  это возможно;
- скорость для первой точки сегмента невычислима из самого сегмента. Код
  чтения, которому доступен только один сегмент, оставляет разрыв. Если
  предыдущий сегмент доступен, код чтения может использовать его последнюю
  точку. Перенос
  `carry-over` значений предыдущего сегмента внутрь сегмента запрещён: это
  скрытая межсегментная зависимость.

Правила для L-колонок:

- label участвует в идентичности или объясняет сущность;
- текстовые label всегда хранятся через `str_id`;
- изменение label без изменения ключа сортировки допустимо только если это
  действительно атрибут, а не часть identity.

## 4. Правила ведения реестра

1. Id выдаётся только добавлением строки в реестр через PR.
2. Id никогда не переиспользуется.
3. Умершие типы помечаются `deprecated`, строки не удаляются.
4. Семантика сбора, ключ сортировки и классификация колонок входят в контракт.
5. Изменение типа колонки, имени колонки, ключа сортировки или семантики сбора
   требует нового `type_id`.
6. Изменение раскладки собственных, не Parquet, блоков всегда требует нового
   `type_id`.
7. Добавление необязательной колонки в конец Parquet-схемы не требует новой версии,
   если это не меняет семантику существующих колонок, старый код чтения
   корректно игнорирует колонку, а новый код чтения трактует её отсутствие как
   `NULL`. Это
   единственное разрешённое совместимое расширение без нового `type_id`.
8. Контракты типов проверяются golden-тестами.
9. `/proc/PID/environ` не имеет и никогда не получит `type_id`.

## 5. Проверки реестра

В CI должен быть registry-linter. Минимальный набор проверок:

- `type_id` уникален и соответствует диапазону класса;
- умершие типы не удалены и помечены `deprecated`;
- все колонки из ключа сортировки существуют в схеме типа;
- в summary-таблице и в детальном описании типа совпадают semantics и ключ
  сортировки;
- у каждого `changed`-типа есть `is_baseline` и описан top-N/coverage-контракт;
- у каждого `on_change`-типа указана политика материализации;
- колонки с `NULL` явно помечены `?` или описаны как необязательные;
- графики используют только допустимые `agg` значения и не смешивают units в
  одной секции графика;
- `values_are` в реестре согласован с `agg`: `rate` требует `agg = 1`,
  `gauge` и `derived` требуют `agg = 0`;
- повтор `type_id` графика в одном сегменте допустим только для разных
  `entity`.

## 6. Self-metrics коллектора

Коллектор должен публиковать собственные метрики, иначе деградация сбора будет
неотличима от нормального отсутствия данных.

Минимальный набор:

- длительность сбора по источнику;
- число строк, прочитанных и записанных по источнику;
- число случаев `dropped`, `timeout` и `permission denied`;
- размер `active.parts` и возраст самого старого мини-сегмента;
- задержка `fdatasync` для мини-сегмента;
- длительность слияния и запечатывания, итоговый размер сегмента;
- число уникальных `str_id`, размер `dict.strings` и `dict.blobs`;
- срабатывания coverage/top-N усечения;
- пропуски регулярного сбора и события `collector_gap`.
